## v1.0.3 smoke matrix — catalog completeness + A13 build-output logging

Manual smoke testing for the v1.0.3 release cycle. Branch: `feat/v1.0.3-catalog-completeness`.

Two surfaces under test:

1. **Catalog completeness** (4 new entries: 3 Lock-Screen / Slideshow CDM values + WindowsStore AutoDownload; commit `8896eeb`) **and the new `registry-pattern-zero` action type** that replaces the 6 hardcoded `SubscribedContent-NNNNNNEnabled` entries with one pattern entry (commit `0a10196`). The pattern action runs in two places: offline build time (mounts NTUSER hive, enumerates matching value names, writes 0 to each) and runtime via the new `Set-RegistryValuePatternToZeroForAllUsers` helper in the post-boot cleanup script.
2. **A13 build-output logging** (commit `e17068a`). GUI Step 1 wizard checkbox + indented "Append to existing log" toggle (default ON / OFF). Headless `tiny11options.exe --log <path>` + `--append` flags (lowercase only; opt-in). File location: `<scratchDir>\tiny11build.log` (GUI) or literal `--log <path>` (headless).

Per the 2026-05-14 binding decision, smoke for both surfaces lives on this branch as the final pre-merge step; v1.0.3 tag + release ceremony gated on green status across the matrix below.

The v1.0.1 P1-P9 baseline (`docs/superpowers/smoke/2026-05-12-post-boot-cleanup-smoke.md`) is the **regression coverage** — re-run all 9 against the new catalog to confirm none of the catalog / pattern-zero / logging changes broke existing behavior. The 6 new cases below are **additive** for the v1.0.3-specific surface.

#### Finding 3 (informational, not a defect): tail-30 of the cleanup log can hide pattern-zero success lines on Core+cleanup-on

The first two runs of `verify-p4-v103-core.ps1` Test 7 reported "no FAKE99 entries in tail-30 of the cleanup log" and the synthesizing read was that pattern-zero hadn't fired for the live user SID. A 2026-05-14 23:22 deep-diagnostic re-run on the same VM showed the opposite: pattern-zero genuinely zeroed FAKE99 (and six real Microsoft-pushed CDM IDs that had accumulated post-boot — `310093`, `338388`, `338389`, `338393`, `353694`, `353696`) end-to-end on the live user SID `S-1-5-21-312255291-2431987326-1127879331-1001`. Decisive evidence:

```
PS> reg query 'HKCU\Software\Microsoft\Windows\CurrentVersion\ContentDeliveryManager' /v 'SubscribedContent-FAKE99Enabled'
HKEY_CURRENT_USER\...\ContentDeliveryManager
    SubscribedContent-FAKE99Enabled    REG_DWORD    0x0    ← was 0x1 immediately before Start-ScheduledTask

PS> Get-Content 'C:\Windows\Logs\tiny11-cleanup.log','C:\Windows\Logs\tiny11-cleanup.log.1' -EA SilentlyContinue |
        Select-String -Pattern 'pattern-zero|FAKE99|SubscribedContent-.*Enabled (CORRECTED|already|correction FAILED)|HKU:\\S-1-5-21'
[2026-05-14 23:22:16]   HKU:\S-1-5-21-312255291-...!SubscribedContent-FAKE99Enabled CORRECTED: '1' -> '0'
[2026-05-14 23:22:16]   HKU:\S-1-5-21-312255291-...!SubscribedContent-310093Enabled = 0 already
[2026-05-14 23:22:16]   HKU:\S-1-5-21-312255291-...!SubscribedContent-338388Enabled = 0 already
... (six pre-existing CDM IDs all reported "= 0 already")
```

Why the prior runs missed it: a single Core+cleanup-on cleanup-task run adds ~216 log lines (verified empirically: 185 → 401 lines in this re-run). Combined with the fact that Core has both Keep-WU-Disabled AND Post-Boot Cleanup tasks plus the daily 03:00 + every-WU-event-19 + 10-min-post-boot triggers, the active log on a Core+cleanup-on VM can grow well past `tail -30` between trigger and grep. The verify-p4 script's Test 7 grep used `Select-Object -Last 3` for the FAKE99-only string match — even narrower. Combined with a likely 15s-wait race (the verify script waits 15s; this re-run measured ~3s between privacy-block start and CDM-block-with-pattern-zero, but a slow Core+cleanup-on first-pass post-boot can push the CDM block past 15s), it's plausible both prior runs missed the success line outright.

**No code change required.** The helper is provably correct in Core+cleanup-on context; the diagnostic recipe used here (snapshot-line-count + plant + trigger + 30s + full-log-grep across `.log` + `.log.1`) is the right tool when re-investigating similar future smoke noise.

**This run is also stronger validation than A11-I3-S2 alone:** A11-I3-S2 only proved synthetic FAKE99 enumeration. Finding 3's diagnostic run additionally proves pattern-zero catches **real** Microsoft-pushed CDM IDs that accumulate at runtime (six of them on this VM). That's the load-bearing guarantee that v1.0.3's `registry-pattern-zero` action type was designed to deliver.

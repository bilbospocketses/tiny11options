## v1.0.3 smoke matrix — catalog completeness + A13 build-output logging

Manual smoke testing for the v1.0.3 release cycle. Branch: `feat/v1.0.3-catalog-completeness`.

Two surfaces under test:

1. **Catalog completeness** (4 new entries: 3 Lock-Screen / Slideshow CDM values + WindowsStore AutoDownload; commit `8896eeb`) **and the new `registry-pattern-zero` action type** that replaces the 6 hardcoded `SubscribedContent-NNNNNNEnabled` entries with one pattern entry (commit `0a10196`). The pattern action runs in two places: offline build time (mounts NTUSER hive, enumerates matching value names, writes 0 to each) and runtime via the new `Set-RegistryValuePatternToZeroForAllUsers` helper in the post-boot cleanup script.
2. **A13 build-output logging** (commit `e17068a`). GUI Step 1 wizard checkbox + indented "Append to existing log" toggle (default ON / OFF). Headless `tiny11options.exe --log <path>` + `--append` flags (lowercase only; opt-in). File location: `<scratchDir>\tiny11build.log` (GUI) or literal `--log <path>` (headless).

Per the 2026-05-14 binding decision, smoke for both surfaces lives on this branch as the final pre-merge step; v1.0.3 tag + release ceremony gated on green status across the matrix below.

The v1.0.1 P1-P9 baseline (`docs/superpowers/smoke/2026-05-12-post-boot-cleanup-smoke.md`) is the **regression coverage** — re-run all 9 against the new catalog to confirm none of the catalog / pattern-zero / logging changes broke existing behavior. The 6 new cases below are **additive** for the v1.0.3-specific surface.

- `Select-String 'RotatingLockScreenEnabled|RotatingLockScreenOverlayEnabled|SlideshowEnabled|AutoDownload' tiny11-cleanup.ps1` → all 4 new v1.0.3 entries present:
  - `Set-RegistryValueForAllUsers ... 'RotatingLockScreenEnabled' ... 0` (line 613)
  - `Set-RegistryValueForAllUsers ... 'RotatingLockScreenOverlayEnabled' ... 0` (line 615)
  - `Set-RegistryValueForAllUsers ... 'SlideshowEnabled' ... 0` (line 617)
  - `Set-RegistryValue -KeyPath 'HKLM:\Software\Policies\Microsoft\WindowsStore' -Name 'AutoDownload' -Type 'DWord' -Value 2` (line 623)

#### Three-run log narrative (proves install, fan-out, idempotency)

- **Run 1 — 13:40:01** (SetupComplete immediate run, pre-login): only `tiny11_default` hive present (no user SID loaded yet). All 4 new CDM values + `AutoDownload=2` reported `already` — proves the **offline build wrote them correctly to the default hive** (offline action paths validated). Pattern-zero on `tiny11_default` reported `has no values matching 'SubscribedContent-*Enabled' (pattern-zero no-op for sid=tiny11_default)` — see Finding 1 below.
- **Run 2 — 17:22:39** (manual `Start-ScheduledTask`, post-login): real user SID `S-1-5-21-2257750218-1729879449-3503378698-1001` AND `tiny11_default` BOTH processed (incidental P7 partial pass — see status table). Real CORRECTIONS observed on user SID:
  - `TailoredExperiencesWithDiagnosticDataEnabled '1' → '0'`
  - `TIPC!Enabled '1' → '0'`
  - `ContentDeliveryAllowed '1' → '0'`
  - `Microsoft-Edge-Webview REMOVED (with takeown)` (proves takeown-and-remove action type works on a populated SystemRoot)
  - `Subscriptions REMOVED` for the user SID (Windows had created the subkey via user activity; cleanup zapped it)
- **Run 3 — 17:42:22** (subsequent run): everything `already` / `absent (no-op)` → **idempotency contract holds**. Pattern-zero confirmed dynamic: still no matching `SubscribedContent-*Enabled` values on the user SID after activity (Microsoft populates server-side over time, not as fresh-install fixtures).

#### Finding 1 (informational, not a defect): pattern-zero finds no `SubscribedContent-*Enabled` values on 25H2

The new `Set-RegistryValuePatternToZeroForAllUsers` helper executes cleanly on every cleanup run but reports `no values matching 'SubscribedContent-*Enabled' (pattern-zero no-op for sid=...)` for both `tiny11_default` and the live user SID. This is **expected behavior on a fresh 25H2 install** — Microsoft no longer pre-populates the `SubscribedContent-NNNNNNEnabled` family in the source ISO's `Users\Default\NTUSER.DAT` (the v1.0.1 catalog's 6 hardcoded IDs were observed-in-the-wild values that Edge/Bing services created over time, not fresh-install fixtures). The mechanism is verified working (clean enumerate-and-skip with informative log entries); proof of dynamic ENUMERATION-AND-ZEROING requires the explicit fabricated-value test (case **A11-I3-S2** below). NOT a regression vs v1.0.1 — v1.0.1 wrote 6 hardcoded entries to the default hive at build time, all of which would have reported `already` on every cleanup run (same observable outcome on a fresh-install hive).

## A11-I3-S2 — Pattern-zero runtime expansion ✅ PASS

- **Date:** 2026-05-14
- **VM:** P1/P2 VM (`v103-p1p2-worker.iso` install)
- **Result:** ✅ PASS — pattern-zero conclusively verified end-to-end on a real machine.
- **Evidence:**
  ```
  PS> reg query 'HKCU\Software\Microsoft\Windows\CurrentVersion\ContentDeliveryManager' /v 'SubscribedContent-FAKE99Enabled'

  HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\ContentDeliveryManager
      SubscribedContent-FAKE99Enabled    REG_DWORD    0x0

  PS> Select-String 'SubscribedContent-FAKE99Enabled' C:\Windows\Logs\tiny11-cleanup.log

  C:\Windows\Logs\tiny11-cleanup.log:750:[2026-05-14 17:56:05]
    HKU:\S-1-5-21-2257750218-1729879449-3503378698-1001\Software\Microsoft\Windows\CurrentVersion\ContentDeliveryManager!SubscribedContent-FAKE99Enabled
    CORRECTED: '1' -> '0'
  ```
- **What this proves:** the `Set-RegistryValuePatternToZeroForAllUsers` helper genuinely enumerates `Get-ItemProperty | Get-Member -MemberType NoteProperty | Where-Object Name -like '<pattern>'` at runtime — not a hidden hardcoded list. A value Microsoft has never shipped (`SubscribedContent-FAKE99Enabled`) was caught and zeroed on the first cleanup-task run after it was planted. This is the dynamic-coverage guarantee that the entire A11-I3 pattern-action work was designed to provide; future Microsoft-added `SubscribedContent-NNNNNNEnabled` IDs will be caught the same way without catalog updates.
- **Closes Finding 1's open question:** earlier cleanup runs found no real `SubscribedContent-*` values to zero on a fresh 25H2 install (Microsoft no longer pre-populates them). S2 verifies the enumeration mechanism works against a real value, eliminating the risk that the empty-set Run 1 / Run 2 / Run 3 results were masking a broken enumerator.

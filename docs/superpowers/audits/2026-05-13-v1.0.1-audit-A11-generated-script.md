# Audit A11: Generated script audit (real catalog)

**Date:** 2026-05-13
**Scope:** Empirical run of `New-Tiny11PostBootCleanupScript` against the real catalog (`catalog\catalog.json`) with every catalog item set to `EffectiveState='apply'`.
**Generated artifact:** `C:\Temp\tiny11-cleanup-real-catalog.ps1` (36,310 chars / 553 lines / 139 top-level statements)
**Parse check:** PS 5.1 parses the generated script cleanly. Zero syntax errors.

## Summary
- BLOCKER: 2 (both confirm A5 findings empirically; one of them is the highest-impact issue in the entire audit)
- WARNING: 2
- INFO: 3

## Methodology

Loaded modules, called `Get-Tiny11Catalog -Path catalog\catalog.json`, fabricated a `ResolvedSelections` hashtable with every item set to `EffectiveState='apply'`, invoked `New-Tiny11PostBootCleanupScript -Catalog $catalog -ResolvedSelections $sel`, wrote output to disk, then parsed and grep-audited it.

## Findings

### BLOCKER

#### [A11-B1] `$env:SystemDrive` and `$env:SystemRoot` are single-quoted in 10 emissions — every filesystem and scheduled-task removal silently no-ops
- **File (generator):** `src/Tiny11.Actions.Filesystem.psm1:51`, `src/Tiny11.Actions.ScheduledTask.psm1:26`
- **Generated script lines:** 382, 384, 386, 390, 400, 430, 434, 438, 442, 446
- **Description:** The filesystem and scheduled-task online-command emitters build path strings like `'$env:SystemDrive\Program Files (x86)\Microsoft\Edge'`. `Format-PSNamedParams` then wraps the value in **single quotes** during emission. PowerShell does NOT expand variables inside single-quoted strings. At runtime the helper receives the literal string `$env:SystemDrive\Program Files (x86)\Microsoft\Edge` and `Test-Path -LiteralPath` returns `$false` for it, so the cleanup function logs "absent (no-op)" and exits.
- **Empirical proof:**
  ```
  PS> $p = '$env:SystemDrive\Program Files (x86)\Microsoft\Edge'
  PS> Test-Path -LiteralPath $p
  False
  ```
  Variable expansion does not happen; the literal string contains a `$` character that NTFS rejects as a path component.
- **Impact:** **The entire filesystem and scheduled-task removal capability is broken across every catalog item that uses these action types.** Edge, EdgeUpdate, EdgeCore, OneDriveSetup.exe, Microsoft-Edge-Webview, and 5 scheduled-task files all fail silently after a CU restages them. Only registry actions (literal `HKLM:\...` keys, no `$env:`) and AppX-prefix removals work as intended.
- **Confirms:** A5 BLOCKER-2 — this audit is the empirical proof for A5's static analysis.
- **Recommended action:** Two surgical options:
  1. Switch the path-string quoting in the emitters from single to double quotes (`"$env:SystemDrive\..."`) so PS expands at runtime. Requires teaching `Format-PSNamedParams` to emit double quotes for strings that begin with `$env:` (or, simpler, for any value that comes from a per-action `[expandable]` shape attribute).
  2. Drop the `$env:` indirection at generation time and emit the resolved `C:\Program Files (x86)\...` literal. Loses one degree of flexibility (works only on the system drive being C:); fine in practice for tiny11.
  Option 2 is simpler and lower-risk; Option 1 preserves more semantic intent.

#### [A11-B2] `HKU:\.DEFAULT` SID fan-out is the wrong target for new-user inheritance
- **File (generator):** `src/Tiny11.PostBoot.psm1:75-78` (header block) and `src/Tiny11.PostBoot.psm1:85-87` (Remove-RegistryKeyForAllUsers)
- **Generated script lines:** 61, 85
- **Description:** Both `Set-RegistryValueForAllUsers` and `Remove-RegistryKeyForAllUsers` enumerate `HKU:\` subkeys matching `S-1-5-21-*` (real user profiles) AND append `'.DEFAULT'`. `HKU:\.DEFAULT` is the registry hive that backs the LOCAL_SERVICE and NETWORK_SERVICE pseudo-profiles. It is NOT the template that fresh user profiles inherit from — fresh profiles inherit from `C:\Users\Default\NTUSER.DAT` which must be `reg load`-ed first.
- **Empirical proof:** All 30+ per-user CDM disablement writes (lines 408, 410, 412, 414, 416, 418, 420, 422, 450, 452, 454, 458, 462, 464, 466, 468, 470, 472, 474, 476, 478, 480, 482, 518, 520, 550 — and the ContentDeliveryManager subkey removes at 488, 490) target `.DEFAULT` which is irrelevant to new user profiles created on the user's machine.
- **Impact:** The whole CDM-disable-for-new-profiles goal stated in the feature design is structurally unmet for any user profile created AFTER the cleanup script runs (i.e., domain joins, second-user accounts, the "first family member" scenario). Existing loaded SIDs are still updated correctly, so single-user machines see correct behavior for the original account. This is the second-most-important finding after A11-B1.
- **Confirms:** A5 BLOCKER-1.
- **Recommended action:** In the helpers block, replace the `'.DEFAULT'` append with a paired `reg load` / write / `reg unload` block targeting `C:\Users\Default\NTUSER.DAT` (loaded into a transient HKU key, e.g. `HKU\tiny11_default`, written, then unloaded). Idempotent guard (skip if already loaded). Document the new-user-profile inheritance semantics in the helper's doc-comment.

### WARNING

#### [A11-W1] `HKU:\` PSDrive may not be auto-mounted in SYSTEM scheduled-task context (PS 5.1)
- **Generated script lines:** 59, 83 (rely on `Get-ChildItem 'HKU:\'`); also lines 514, 516 (direct `HKU:\.DEFAULT` writes); plus all `Set-RegistryValueForAllUsers` / `Remove-RegistryKeyForAllUsers` calls
- **Description:** When `tiny11-cleanup.ps1` runs as the scheduled task under SYSTEM, the PS session may not have an `HKU:` PSDrive auto-mounted. PS 5.1's standard PSDrive auto-mounts are `HKLM:` and `HKCU:`, with `HKU:` historically requiring `New-PSDrive -Name HKU -PSProvider Registry -Root HKEY_USERS`. If it's not mounted, `Get-ChildItem 'HKU:\'` returns nothing AND every `Set-RegistryValue -KeyPath "HKU:\..."` fails (Test-Path returns false). The function logs `"...absent (no-op)"` and skips.
- **Confirms:** A7 WARNING (cited as part of the WinSxS/SYSTEM-context audit).
- **Recommended action:** Add a defensive `if (-not (Get-PSDrive -Name HKU -ErrorAction SilentlyContinue)) { New-PSDrive -Name HKU -PSProvider Registry -Root HKEY_USERS | Out-Null }` near the top of the helpers block. Cheap, idempotent, fixes silent failure mode.

#### [A11-W2] Task XML URI uses non-standard root namespace
- **File:** `src/Tiny11.PostBoot.psm1:242` — `<URI>\tiny11options\Post-Boot Cleanup</URI>`
- **Description:** Many enterprise tooling stacks expect scheduled tasks under `\Microsoft\` or a vendor-prefixed namespace, not a top-level `\tiny11options\`. The breadcrumb's prior session notes reference `\Microsoft\tiny11options Post-Boot Cleanup` — that's inconsistent with what the code actually emits. Both the XML's `<URI>` and the `SetupComplete.cmd` schtasks `/tn` argument use `tiny11options\Post-Boot Cleanup` (no Microsoft prefix); internally consistent but probably not what the design intended.
- **Recommended action:** Pick a convention and document it. Either `\Microsoft\tiny11options\Post-Boot Cleanup` (matches breadcrumb intent + enterprise convention) or stay with `\tiny11options\Post-Boot Cleanup` (advertised as vendor-owned task tree). Update both the XML and the SetupComplete.cmd to match the chosen convention.

### INFO

#### [A11-I1] Generated script parses cleanly in PS 5.1
- 553 lines, 139 top-level statements, 0 syntax errors via `[System.Management.Automation.Language.Parser]::ParseFile()`.

#### [A11-I2] Helpers block uses `New-Item -Path` (not `-LiteralPath`) for registry key creation
- **Generated script line:** 41 (inside `Set-RegistryValue`)
- A registry key path containing `[`, `]`, or other PS wildcard characters would be misinterpreted by `New-Item -Path`. `-LiteralPath` would be safer. In practice the catalog's emitted keys are all literal Windows paths with predictable shapes, so this is a latent rather than active fault. Worth fixing as a defensive nicety.

#### [A11-I3] `SubscribedContent-*Enabled` removal coverage is hardcoded
- **Generated script lines:** 470–480 — hardcoded SubscribedContent IDs (310093, 338388, 338389, 338393, 353694, 353696).
- The breadcrumb's design notes say "remove all subkeys / properties matching `SubscribedContent-*Enabled`". The implementation hardcodes a finite list of IDs known at design time. New IDs that Microsoft adds in future CUs (which is the whole point of having a recurring cleanup task) will not be covered. Consider replacing with a pattern-driven enumeration: `Get-ItemProperty HKU:\<sid>\Software\...\ContentDeliveryManager | Get-Member -MemberType NoteProperty | Where-Object Name -like 'SubscribedContent-*Enabled' | ForEach-Object { Set-ItemProperty ... 0 }`.

## Executive summary

**The single highest-impact finding is [B1] `$env:SystemDrive` single-quoting** — every filesystem-removal and scheduled-task-removal action in the generated cleanup script silently no-ops because PowerShell does not expand variables inside single-quoted strings. Verified empirically: `Test-Path -LiteralPath '$env:SystemDrive\Program Files (x86)\Microsoft\Edge'` returns `False`. This means Edge, OneDrive, WebView, and 5+ scheduled-task removals all fail silently after every CU. Only registry actions and AppX-prefix removals work as intended on the generated script.

The audit also surfaced **3 additional module-scope landmines in `Core.psm1`** with the same cascade-shadow signature as the regression at `bdb38d7`. Core-mode smoke testing would have hit at least one of these. The structural fix is **a single line**: add `-Global` to `src/Tiny11.Actions.Registry.psm1:2`'s Hives import. That one change makes all four scope-related BLOCKERs structurally safe.

The audit also caught a **silent Velopack-update-chain break** (csproj missing `<Version>`), **two regression-coverage gaps** (no test exists for either of the two regressions we just hit), and a **misleading CHANGELOG entry** that claims a fix where none was applied.

**Recommended:** apply the 5 critical BLOCKER fixes (B1-B5 below) before resuming the smoke matrix; the other 9 can be triaged into either pre-smoke or follow-up.

## Critical BLOCKERs (recommended pre-smoke fixes)

These 5 are the must-fix-before-P1d items. Without these fixed, smoke testing burns 30–45 minutes per iteration to surface known regressions.

### [B1] `$env:SystemDrive` / `$env:SystemRoot` are single-quoted in generated script → filesystem + scheduled-task removals silently no-op

- **Source:** `src/Tiny11.Actions.Filesystem.psm1:51`, `src/Tiny11.Actions.ScheduledTask.psm1:26`
- **Generated lines affected:** 382, 384, 386, 390, 400, 430, 434, 438, 442, 446 (Edge x3, OneDriveSetup.exe, Microsoft-Edge-Webview, 5 scheduled-task files)
- **Found by:** A5 (static analysis) + A11 (empirical confirmation)
- **Root cause:** `Format-PSNamedParams` wraps string values in single quotes; the emitters pass paths containing `$env:SystemDrive` as ordinary strings; PS does not expand `$env:` inside single quotes.
- **Empirical proof:** `Test-Path -LiteralPath '$env:SystemDrive\Program Files (x86)\Microsoft\Edge'` → `False`.
- **Impact:** **The entire filesystem + scheduled-task removal capability of the post-boot cleanup task is non-functional across the entire catalog.** Only registry actions and AppX removals actually do anything.
- **Fix direction (lowest risk):** Resolve `$env:` paths at generation time and emit literal `C:\Program Files (x86)\...` strings. The build runs on the user's machine, `$env:SystemDrive` is `C:` in 99.9%+ of real installs, and this loses no functionality the user actually depends on.
- **Fix direction (preserves intent):** Teach `Format-PSNamedParams` to emit double-quoted strings (or `[expandable]`-typed values) so `$env:` expands at runtime. More flexible but higher-risk because it changes the quoting story for every string.

### [B2] Core mode reproduces module-scope Regression 2 because `Actions.Registry.psm1:2` lacks `-Global` on its Hives import

- **Source:** `src/Tiny11.Actions.Registry.psm1:2` (structural root cause, marked I1 in A1's report); cascades fire at `src/Tiny11.Core.psm1:14`, `:1095`, `:1452`
- **Found by:** A1
- **Description:** `Actions.Registry.psm1`'s `Import-Module Tiny11.Hives.psm1 -Force` has no `-Global`. Any caller that reloads Actions with `-Force` (including PostBoot, Worker top-level, Core top-level, mid-pipeline Worker reload at Core:1452) cascades through Actions → Actions.Registry → Hives and demotes Hives from global scope. The original `bdb38d7` was a band-aid that only protected the PostBoot caller; the structural defect was never fixed.
- **Specific Core-mode failures predicted:**
  - **A1-B1** (`Core.psm1:14`): `Import-Module PostBoot -Force -Global` re-enacts the cascade in Core's first import.
  - **A1-B2** (`Core.psm1:1095`): mid-pipeline defensive `Import-Module Hives -Force` (no `-Global`) reloads at function scope only; defensive intent is correct, implementation is wrong.
  - **A1-B3** (`Core.psm1:1452`): mid-pipeline `Import-Module Worker -Force` (no `-Global`) re-executes Worker's body, which re-imports PostBoot with `-Force -Global`, which cascades the chain again, demoting Hives at percent=98.
- **Single-line structural fix:** add `-Global` to `Actions.Registry.psm1:2`'s Hives import. Then audit the other Actions.* modules for the same pattern (likely all four should have `-Global`).
- **Impact without fix:** Core-mode smoke testing will hit at least one of the three landmines.

### [B3] `launcher/tiny11options.Launcher.csproj` has no `<Version>` → Velopack update chain breaks silently from day one of v1.0.1 release

- **Source:** `launcher/tiny11options.Launcher.csproj` (no `<Version>` / `<AssemblyVersion>` / `<FileVersion>` in PropertyGroup)
- **Found by:** A2-B1, A9-B1 (same finding from two angles)
- **Description:** `app.manifest` declares `version="1.0.1.0"` (Win32 assembly identity, used by Windows for side-by-side resolution and DPI/asInvoker semantics) but the csproj has no `<Version>` property, so MSBuild auto-generates `1.0.0.0` for all `[assembly: ...]` attributes AND the NuGet package version. Velopack reads the NuGet version. Result: v1.0.1 ships as Velopack package version `1.0.0`, the auto-update chain sees no upgrade for existing v1.0.0 users, and Velopack treats v1.0.1 as a re-publish of v1.0.0.
- **Also breaks:** `MainWindow.ResolveUiCacheDir()` and `ResolveResourcesDir()` cache paths key on `Assembly.GetExecutingAssembly().GetName().Version`, so v1.0.1 reuses the v1.0.0 cache directory. Hash-based extraction still rescues content correctness but the version mismatch is wrong on its face and will confuse Velopack release manifests + Windows Explorer File Properties.
- **Fix:** add `<Version>1.0.1</Version>` to the csproj `<PropertyGroup>`. One line. Test: rebuild and inspect `dist/raw/tiny11options.exe` properties — should report 1.0.1.0.

### [B4] `HKU:\.DEFAULT` is the wrong target for new-user CDM-inheritance — feature design goal structurally unmet for new user profiles

- **Source:** `src/Tiny11.PostBoot.psm1:75-78` and `:85-87` (helpers block)
- **Found by:** A5-B1 (static) + A11-B2 (empirical confirmation across 30+ emitted lines)
- **Description:** `HKU:\.DEFAULT` backs LOCAL_SERVICE / NETWORK_SERVICE pseudo-profiles, NOT the new-user inheritance template. New profiles inherit from `C:\Users\Default\NTUSER.DAT`, which must be `reg load`-ed first.
- **Impact:** Existing loaded SIDs (the current user, already-loaded sessions) ARE correctly updated. New user accounts created post-cleanup (domain join, additional family member, fresh sysprep) do NOT inherit the CDM-disablement settings. The "CDM persistence across profiles" design goal is structurally broken for these cases.
- **Fix direction:** In the helpers block, replace `$sids += '.DEFAULT'` with a paired `reg load HKU\tiny11_default 'C:\Users\Default\NTUSER.DAT'` + write + `reg unload HKU\tiny11_default` block. Idempotent guard (skip-if-already-loaded). Document the new-user-profile inheritance semantics inline.

### [B5] `$args` automatic-variable shadowing in `Tiny11.Core.psm1:499` (`Invoke-CoreIcacls`)

- **Source:** `src/Tiny11.Core.psm1:499`
- **Found by:** A7
- **Description:** Local variable named `$args` shadows the PS automatic variable. Under `Set-StrictMode -Version Latest` this is fragile and version-dependent. If the assignment misfires, `icacls.exe` runs without args; subsequent `Remove-Item` fails with access-denied in the WinSxS wipe, filesystem-removal, and boot.wim prep phases.
- **Fix:** Rename to `$icaclsArgs` (matches the `$takeownArgs` name in `Invoke-CoreTakeown` directly above). One-line fix.

## Resilience BLOCKERs (recommended fix before v1.0.1 release tag)

These do not block smoke testing directly but mask root causes during failures and degrade UX. Fix before tagging v1.0.1.

### [B6] Worker pipeline leaves WIM mounted when `Install-Tiny11PostBootCleanup` throws

- **Source:** `src/Tiny11.Worker.psm1:112,115,155-164` (`Invoke-Tiny11BuildPipeline`)
- **Found by:** A4-B1
- **Description:** Install is called at line 112 BEFORE `Dismount-WindowsImage -Save` at line 115. The `finally` block (155-164) dismounts the source ISO and deletes scratch dirs but does NOT call `Dismount-WindowsImage`. On Install failure the mount is abandoned. `Remove-Item -ErrorAction SilentlyContinue` on the scratch dir then silently fails, leaving a dangling mount that requires manual `dism /Cleanup-Mountpoints`.
- **Reference:** Core has this pattern correct (commit / discard via `$pipelineSucceeded` flag at Core.psm1:1155,1336,1341).
- **Fix:** Port Core's commit/discard pattern to Worker. Add a `$pipelineSucceeded` flag at the top of the try block; on the finally branch, `Dismount-WindowsImage -Discard` if the flag is false.

### [B7] Worker `Install-Tiny11PostBootCleanup` exception is not surfaced with phase context

- **Source:** Same — `Worker.psm1:112`
- **Found by:** A4-B2
- **Description:** Install is called bare with no try/catch and no phase marker; an exception propagates as raw text without telling the caller which step failed. Core handles this correctly via its `$pipelineSucceeded` + phase-marker pattern.
- **Fix:** Wrap the call in a try/catch that emits `Write-Marker 'pipeline-phase' 'install-post-boot-cleanup'` before invocation and re-throws on failure. Or port the Core pattern wholesale (preferred).

### [B8] `Remove-AppxByPackagePrefix` uses Appx cmdlets without `Import-Module Appx`

- **Source:** `src/Tiny11.PostBoot.psm1:141-170` (helpers block)
- **Found by:** A5-B3
- **Description:** Four Appx cmdlets used in the helper (`Get-AppxProvisionedPackage`, `Remove-AppxProvisionedPackage`, `Get-AppxPackage`, `Remove-AppxPackage`). On Win11 desktop sessions auto-discovery handles this. On a debloated image or any SYSTEM-context PS 5.1 scheduled-task session where auto-discovery is impaired, the cmdlets are not found and the function throws "not recognized."
- **Fix:** Add `Import-Module Appx -ErrorAction SilentlyContinue` (or `-ErrorAction Stop` if we want to surface failure) at the top of `Remove-AppxByPackagePrefix` (or once at the top of the helpers block). Cheap, idempotent, defensive.

## Regression-coverage gaps (recommended fix before resuming smoke)

These two tests are missing — the regressions we just hit could land again on the next refactor with no warning.

### [B9] No xUnit test asserts `Tiny11.PostBoot.psm1` is in the launcher's embedded-resource glob (regression `b35d73a` coverage)

- **Source:** `launcher/Tests/EmbeddedResourcesTests.cs` — `RealResource_IsEmbedded` is missing `[InlineData("src/Tiny11.PostBoot.psm1")]`
- **Found by:** A6-B1
- **Description:** Existing test asserts the embedded-resource glob covers `Tiny11.Worker.psm1` and other modules but not PostBoot. The csproj glob silently dropped PostBoot, then HeadlessRunner's hardcoded `StaticResources[]` also lacked it — both were necessary for the regression to ship.
- **Fix:** Add `[InlineData("src/Tiny11.PostBoot.psm1")]`. Also add a sister assertion that `HeadlessRunner.StaticResources` matches the csproj `<EmbeddedResource>` list (the actual structural assertion — the drift between them is what caused `b35d73a`).

### [B10] No Pester test runs `Worker → Actions → PostBoot` end-to-end and asserts `Mount-Tiny11AllHives` is callable (regression `bdb38d7` coverage)

- **Source:** No covering test exists
- **Found by:** A6-B2
- **Description:** Existing Worker tests assert post-boot wiring exists by regex-matching source text (`Get-Content -Raw`). This cannot detect runtime scope demotion. A test that loads Worker → triggers full PostBoot import chain → asserts `Get-Command Mount-Tiny11AllHives -All | Where-Object Source -eq 'Tiny11.Hives'` returns the global-scope instance would catch this class of regression.
- **Fix:** Add `tests/Tiny11.Worker.PostBootImport.IntegrationTests.ps1` that performs the full chain load and asserts global-scope visibility of Hives commands. Pairs naturally with the [B2] structural fix.

## Selected high-value WARNINGs (full list in per-category files)

These are the WARNINGs most worth surfacing during triage; consult the per-category files for the rest.

### Test isolation (A6)

- **A6-W (Hives flakies root cause):** `Tiny11.TestHelpers.psm1` uses `-Force -Global`. v1.0.1 added **12 new** test files that use standalone `Import-Module -Force` (no `-Global`). Each standalone import demotes the global instance pending mocks were bound to. `Should -Invoke -ModuleName 'Tiny11.Hives'` then targets the stale instance and fails. v1.0.1 multiplied collision opportunities. **Pair with the [B2] structural fix** — if Actions.Registry's import gains -Global, this likely heals too.
- **Pester version not pinned** — floor-checked `>= 5.0.0`; Pester 5.x has had mock-scoping behavioral changes.
- **Golden fixture is byte-equal compared** with no line-ending normalization — Windows checkout that CRLFs the fixture spuriously fails.

### Generated script (A11) and PostBoot internals (A5)

- **A11-W1 / A7:** `HKU:\` PSDrive not auto-mounted in SYSTEM scheduled-task context on PS 5.1 — every `Set-RegistryValueForAllUsers` / `Remove-RegistryKeyForAllUsers` silently no-ops. Add `New-PSDrive -Name HKU -PSProvider Registry -Root HKEY_USERS` guard at top of helpers block.
- **A11-W2:** Task XML URI uses `\tiny11options\Post-Boot Cleanup` (non-standard root) — internally consistent but probably not the design intent. Pick a convention.
- **A5-W:** Missing `RestartOnFailure` in task XML. Unknown action type terminates generator silently. REG_MULTI_SZ idempotency check broken via array `-eq` semantics. Empty `packagePrefix` would nuke all Appx.

### Wrappers (A3)

- Mixed `[bool]` + `[switch]` param pattern is asymmetric with all other toggle params (pure `[switch]`) and produces fragile self-elevation serialization. No production breakage, but a refactoring footgun.
- No mutual-exclusivity guard between `-InstallPostBootCleanup $true` and `-NoPostBootCleanup`. Passing both silently resolves to false. Deterministic but undocumented.
- Help block doesn't document `-InstallPostBootCleanup` / `-NoPostBootCleanup` — Get-Help users can't discover the opt-out.

### Pipeline (A4)

- Core's `Install-Tiny11CorePostBootCleanup` silently no-ops cleanup files when `$PostBootCleanupCatalog` is `$null` even when `$PostBootCleanupEnabled = $true`. No warning emitted.
- Neither pipeline removes previously-installed artifacts when `$Enabled = $false`. Re-building with cleanup disabled over a prior enabled build leaves stale files in the WIM.
- Core's SetupComplete.cmd only *registers* the cleanup task via `schtasks /create`; Worker's version also runs `tiny11-cleanup.ps1` immediately — intentional but undocumented asymmetry.

### Style (A7)

- `Invoke-Takeown`, `Invoke-Icacls`, `Get-ProvisionedAppxPackagesFromImage`, `Invoke-DismRemoveAppx` lack `[CmdletBinding()]`.
- `Format-PSNamedParams` exported despite being internal helper.

### Documentation (A8 + A9)

- CHANGELOG xUnit count wrong: "80 → 82" but actual diff adds 4 facts → should be "80 → 84."
- P1-smoke regression-fix entries landed in `[Unreleased] ### Fixed` rather than `## [1.0.1]` — pre-release move them.
- Plan smoke tasks 29-36 never annotated; `docs/superpowers/smoke/2026-05-12-post-boot-cleanup-smoke.md` was never created per plan Task 29 Step 5.

### A13 (headless log redirection)

- **Root cause** of the smoke-debugging pain: `WinExe` subsystem + lazy-initialized `Console.Out` not refreshed after `AttachConsole(-1)` + child `pwsh` spawned with `RedirectStandardOutput = false`. Three layers of failure mode; current `ConsoleAttach.AttachToParent()` swallows the attach failure silently.
- **Recommended quick win** (~45 min, unblocks smoke): add `--log <path>` flag; set `RedirectStandardOutput = true`; async-read stdout and fan to both `Console.Out` and a log `StreamWriter`. ~75-100 lines, one new test.
- **Recommended right answer** (~20-30 lines, later): fix `ConsoleAttach` to check return value, call `Console.SetOut(new StreamWriter(Console.OpenStandardOutput()))` after a successful attach, fall back to `AllocConsole()` when attach fails (the piped-tee case). This is the `git.exe`/`dotnet.exe` pattern.

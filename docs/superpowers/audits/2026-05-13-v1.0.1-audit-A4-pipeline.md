## Findings

### BLOCKER-1: Worker error path — WIM stays mounted when Install throws

**File:** `src/Tiny11.Worker.psm1` lines 54-164

The outer `try` block in `Invoke-Tiny11BuildPipeline` has a `finally` that cleans up scratch directories (`$tinyDir`, `$scratchImg`) — **but no `Dismount-WindowsImage` call is in that `finally`**. The only `Dismount-WindowsImage -Save` call is at line 115, inside the `try` body, AFTER `Install-Tiny11PostBootCleanup` at line 112. If `Install-Tiny11PostBootCleanup` throws (e.g., `New-Item` fails, disk full, permission error), execution jumps to the `finally` at line 155 which:

- Dismounts the **source ISO** (line 157) — correct
- Deletes `$tinyDir` and `$scratchImg` directories (lines 162-163) — but the WIM is still mounted inside `$scratchImg` at this point

Deleting a directory with a mounted WIM image inside it fails silently (`-ErrorAction SilentlyContinue`) and leaves the mount dangling. The user sees no `Dismount-WindowsImage` error; they get a corrupted/stale mount. Recovery requires `dism /Cleanup-Mountpoints`.

**Severity: BLOCKER** — produces a dangling WIM mount with no user guidance on recovery.

---

### BLOCKER-2: Worker error path — no explicit error surfaced when Install throws

**File:** `src/Tiny11.Worker.psm1` lines 111-113

`Install-Tiny11PostBootCleanup` is called bare — no `try/catch` wrapping it, no progress marker on failure. If the function throws, the exception propagates directly to the outer `try`'s `finally` (which doesn't log it), then up to the caller. The caller (`Tiny11.Bridge.psm1` or wrapper script) receives a raw PS exception. There is no phase-marker emitted to indicate the failure was in the Install step specifically, making diagnosis harder for the launcher's error UI.

Compare with Core (BLOCKER-3 below for a different issue there): Core at least sets `$pipelineSucceeded = $false` implicitly by not reaching line 1336, which gates the `/Commit` vs `/Discard` decision.

**Severity: BLOCKER** — combined with BLOCKER-1, the user gets a dangling mount AND an opaque error with no recovery path shown.

---

### BLOCKER-3: Core — Install fires OUTSIDE the `$pipelineSucceeded` guard try-block

**File:** `src/Tiny11.Core.psm1` lines 1331-1337

The sequence is:

```
Install-Tiny11CorePostBootCleanup ...   # line 1331-1334
$pipelineSucceeded = $true              # line 1336
```

These two lines are INSIDE the `try` block (lines 1156-1337). If `Install-Tiny11CorePostBootCleanup` throws after writing some files but before completing all three (or four, when cleanup is enabled), `$pipelineSucceeded` stays `$false`. The outer `finally` (line 1338) then calls `Invoke-CoreDism /Discard`, which correctly discards the partially-written WIM. This is better than Worker's behavior.

**HOWEVER:** `Install-Tiny11CorePostBootCleanup` at line 951-1001 calls `New-Tiny11CorePostBootCleanupScript` then `New-Tiny11PostBootCleanupScript` then `New-Tiny11CoreWuEnforceScript` etc. None of the individual `[System.IO.File]::WriteAllText` calls are wrapped — a partial write (disk full mid-way through writing the 4-file set) produces a partially-installed artifact set that the `/Discard` then throws away. That's correct behavior. The issue is that the phase marker at line 1330 claims "Installing SetupComplete.cmd for first-boot /Cleanup-Image" which doesn't mention the v1.0.1 cleanup files — a user reading the log can't tell which file write failed.

**Severity: BLOCKER** — not a correctness blocker (Core's /Discard logic is sound), but the phase marker misidentifies the work scope. Re-classified: **WARNING** on closer reading (see WARNING-1 below). Demoting BLOCKER-3.

**Revised: BLOCKER-3 is demoted to WARNING-1.** The actual blocker count is 2.

### WARNING-2: Worker — no `$pipelineSucceeded` guard / no `/Discard` on failure

**File:** `src/Tiny11.Worker.psm1` lines 112-115

Worker uses the DISM Cmdlet (`Dismount-WindowsImage -Save`) not the CLI, and only calls it in the happy path. Core has an explicit `$pipelineSucceeded` flag and selects `/Commit` vs `/Discard` in `finally`. Worker has no equivalent guard — if Install throws, the WIM is never dismounted at all (BLOCKER-1), and if some later step throws after Install succeeds, the WIM is similarly left mounted. Core's pattern here is strictly better.

**Severity: WARNING** (the mount-leak is BLOCKER-1; this is the root cause pattern)

### WARNING-4: Worker — `Install-Tiny11PostBootCleanup` has `[bool] $Enabled = $true` as non-Mandatory, with no null/false audit trail

**File:** `src/Tiny11.PostBoot.psm1` lines 308-316

```powershell
param(
    [Parameter(Mandatory)][string]    $MountDir,
    [Parameter(Mandatory)]            $Catalog,
    [Parameter(Mandatory)][hashtable] $ResolvedSelections,
    [bool]                            $Enabled = $true
)

```

When `$Enabled = $false`, the function returns immediately with no log output, no progress marker, and no artifact cleanup. If a prior build run had installed the files (idempotency scenario — see INFO-1), and the user rebuilds with `$InstallPostBootCleanup = $false`, the WIM will contain stale v1.0.1 artifacts. The function does not remove previously-written files. This is a latent idempotency gap for the "disable after prior enable" scenario.

**Severity: WARNING**

### INFO-1: Idempotency — re-run over an image that already has the artifacts

**Files:** `src/Tiny11.PostBoot.psm1` lines 318-343; `src/Tiny11.Core.psm1` lines 960-999

Both `Install-Tiny11PostBootCleanup` (Worker) and `Install-Tiny11CorePostBootCleanup` (Core) use `[System.IO.File]::WriteAllText`, which unconditionally overwrites any existing file at the path. `New-Item -ItemType Directory -Force` on the parent is idempotent. So if called twice against the same mounted image:

- The 3 (Worker) or 4 (Core, when enabled) files are overwritten cleanly.
- No stale files are left from the prior write.
- No failure.

**Verdict: overwrite-idempotent for the $true -> $true case.** The $true -> $false case (WARNING-4) is NOT idempotent.

### INFO-2: Conditional logic — `$InstallPostBootCleanup` parameter behavior

**Worker** (`src/Tiny11.Worker.psm1` line 45): `[Parameter()][bool]$InstallPostBootCleanup = $true`

When `$true`: Install fires, files written.
When `$false`: `Install-Tiny11PostBootCleanup -Enabled:$false` fires at line 112, but the function returns immediately at line 316 — no files written, no artifacts from prior runs removed.
When `$null`/unset: PowerShell coerces unset `[bool]` to `$false` at binding time if the caller omits it; the default `$true` applies if not overridden. Unset callers get cleanup installed by default. Correct behavior.

**Core** (`src/Tiny11.Core.psm1` line 1089): `[bool]$InstallPostBootCleanup = $true` (no `[Parameter(Mandatory)]`)

Same semantics. The three-parameter call at line 1331-1334 passes `$InstallPostBootCleanup` as `-PostBootCleanupEnabled`. When `$null` (unset), same default-$true behavior.

### INFO-3: Catalog + ResolvedSelections threading

**Worker** (`src/Tiny11.Worker.psm1` lines 34-46, 112):

`$Catalog` and `$ResolvedSelections` are declared `[Parameter(Mandatory)]` on `Invoke-Tiny11BuildPipeline`. They are passed directly to `Install-Tiny11PostBootCleanup` at line 112 without modification. `$ResolvedSelections` is typed `[hashtable]` — passed by reference in PowerShell. `$PostBootCleanupCatalog` and `$PostBootCleanupResolvedSelections` are passed through from pipeline params to `Install-Tiny11CorePostBootCleanup` unmodified. No mutation between receipt and use.

`$scriptDir = Join-Path $MountDir 'Windows\Setup\Scripts'` where `$MountDir` = `$mountDir` = `Join-Path $ScratchDir 'mount'`. Same safe offline-mount path. No host PROGRAMDATA risk.

### INFO-5: SetupComplete.cmd generator integration (Core)

**File:** `src/Tiny11.Core.psm1` lines 637-706

`New-Tiny11CorePostBootCleanupScript` accepts `-IncludePostBootCleanupRegistration` (switch). When `$true`, it appends the `$cleanupBlock` section:

```cmd
schtasks /create /xml "%SystemDrive%\Windows\Setup\Scripts\tiny11-cleanup.xml" /tn "tiny11options\Post-Boot Cleanup" /f
```

This registers the scheduled task via the XML. The XML (`tiny11-cleanup.xml`) and script (`tiny11-cleanup.ps1`) are both written to the same `Scripts` directory by `Install-Tiny11CorePostBootCleanup` when `$writeCleanupFiles = $true`. The task XML embeds an absolute path `C:\Windows\Setup\Scripts\tiny11-cleanup.ps1` (from `New-Tiny11PostBootTaskXml` in PostBoot.psm1 line 278).

No SetupComplete.cmd line runs `tiny11-cleanup.ps1` directly — that's left to the scheduled task's BootTrigger (PT10M delay). The SetupComplete.cmd only registers the task via `schtasks /create`, which is correct.

**Worker** uses a different SetupComplete.cmd (`New-Tiny11PostBootSetupCompleteScript` in PostBoot.psm1 lines 285-306). That script both registers the task via `schtasks /create /xml` AND immediately runs `tiny11-cleanup.ps1` once directly (lines 300-301). Core's SetupComplete.cmd does NOT run cleanup.ps1 directly — it relies on the BootTrigger. This asymmetry is **intentional** per the comments but is worth noting.

## Root cause analysis

### Subsystem: WinExe (confirmed)

`launcher/tiny11options.Launcher.csproj:4` — `<OutputType>WinExe</OutputType>`

This is the Windows GUI subsystem. The OS does not create a console window when the EXE is
launched from Explorer, and `Console.Out`/`Console.Error` are backed by null handles unless a
console is explicitly attached via the Win32 API.

### The AttachConsole partial fix (already exists)

`launcher/Interop/ConsoleAttach.cs` already implements `AttachConsole(ATTACH_PARENT_PROCESS)`
and `FreeConsole()`. `HeadlessRunner.Run()` (line 35) calls `ConsoleAttach.AttachToParent()`
as its very first action, before any Console writes or child-process spawning.

This _should_ work — and in a normal console window (cmd.exe or Windows Terminal) it likely
does. The `ATTACH_PARENT_PROCESS` (-1) flag asks the OS to wire the EXE's stdout/stderr to
the parent console's handles.

### Why `| Tee-Object` and `> file.log` fail anyway

This is the crux. There are two distinct failure modes:

**Failure mode 1 — Pipe redirection (`| Tee-Object`, `> out.log`)**

When pwsh redirects the EXE's output (`tiny11options.exe ... | Tee-Object`), the parent
process does not own a real console — pwsh has replaced its own stdout handle with a pipe.
`AttachConsole(ATTACH_PARENT_PROCESS)` attaches to the **console**, not to the pipe. The Win32
contract for `AttachConsole` is:

> "Attaches the calling process to the console of the specified process."

If the parent process has no console (because it is itself piped), `AttachConsole` returns
`FALSE` (error `ERROR_INVALID_HANDLE` or `ERROR_GEN_FAILURE`). The current implementation
ignores the return value (comment: "Best-effort ... Failures are non-fatal").

After a failed `AttachConsole`, `Console.Out` still points at null handles. The child
`powershell.exe` process is then launched with `RedirectStandardOutput = false` and
`RedirectStandardError = false` (HeadlessRunner.cs:58-59), meaning the child inherits the
EXE's stdout handle — which is still null. **All child output is silently discarded.**

**Failure mode 2 — Console window but wrong console**

Even when `AttachConsole` succeeds (user runs from a normal terminal without piping),
`Console.Out` and `Console.Error` re-use the pre-existing `TextWriter` instances that were
set up before attach. On .NET, `Console.Out` is lazily initialized once; calling
`AttachConsole` after the process starts does not automatically refresh the managed
`Console.Out` to point at the new console handle. A call to `Console.OpenStandardOutput()`
and `Console.SetOut(new StreamWriter(...))` is required after attach. The current code does
not do this, so `Console.Error.WriteLine(...)` calls at HeadlessRunner.cs:46 and 73 may
silently drop even in the non-piped case.

The child process (`powershell.exe`) inherits file handles at `Process.Start` time.
`UseShellExecute = false` causes the child to inherit the EXE's stdin/stdout/stderr handles
(the Win32 `STARTUPINFO.hStdOutput` is copied). If those handles are null at start time,
the child's stdout is also null — it writes to nowhere regardless of what `-WindowStyle
Hidden` does.

### Child process stdout handling

`HeadlessRunner.cs:53-62`:
```
var psi = new ProcessStartInfo
{
    FileName = "powershell.exe",
    ...
    UseShellExecute = false,
    RedirectStandardOutput = false,   // inherit from attached console
    RedirectStandardError = false,
    CreateNoWindow = true,
};
```

`RedirectStandardOutput = false` with `UseShellExecute = false` means the child inherits the
EXE's stdout handle directly. This is the correct design intent — "let the child write
directly to wherever the launcher's console is" — but it only works when `AttachConsole`
succeeded AND `Console.SetOut` was refreshed after attach.

When the user runs `tiny11options.exe ... | Tee-Object out.log`:
1. pwsh allocates a pipe; EXE's inherited stdout = the pipe write end.
2. EXE starts; it is a `WinExe` — OS does not create a console.
3. `AttachConsole(-1)` is called. The parent (pwsh) has no console (it is piped). `AttachConsole` returns FALSE.
4. `Console.Out` / `Console.Error` remain pointed at null.
5. `powershell.exe` is spawned. It inherits stdout = null handle from EXE.
6. All child writes go to null. `Tee-Object` receives EOF immediately. `out.log` is empty.

Note: the EXE's initial inherited stdout from pwsh _is_ the pipe write end, but `Console.Out`
was never refreshed to use it after (the failed) `AttachConsole`. Had `AttachConsole` been
skipped entirely and `Console.SetOut` used to re-open stdout from `Console.OpenStandardOutput()`,
the pipe handle might have been usable. This is nuanced and empirically unverified — marked
UNCERTAIN. The safe fix is Option B or C below.

## Recommendation

**Short-term (unblocks v1.0.1 smoke testing): implement Option B — `--log <path>` flag.**

Scope estimate:
- `launcher/Program.cs` or a new `launcher/Headless/HeadlessArgs.cs`: add `--log` arg parsing (~15 lines)
- `launcher/Headless/HeadlessRunner.cs`: change `RedirectStandardOutput = true` + `RedirectStandardError = true`; add async stdout reader that writes to both `Console.Out` and the log `StreamWriter` (~40 lines)
- `launcher/Tests/HeadlessRunnerTests.cs`: add 1-2 unit tests for arg parsing + the forwarder callback shape (~20 lines)
- No changes to `.csproj`, `ConsoleAttach.cs`, or any PS scripts

Total: ~3-4 files, ~75-100 lines, est. 30-45 minutes.

**Long-term (right answer): add Option C on top of B.**

Fix `ConsoleAttach.AttachToParent()` to:
1. Check the return value of `AttachConsole`.
2. If it succeeded, call `Console.SetOut(new StreamWriter(Console.OpenStandardOutput()) { AutoFlush = true })` and the equivalent for `Console.Error`.
3. If it failed (parent has no console — i.e., piped scenario), call `AllocConsole()` as fallback, then do the same `SetOut` refresh.

This makes `| Tee-Object` work natively without `--log`. It is what `git.exe`, `dotnet.exe`,
and `winget.exe` all do. Scope: ~20-30 lines in `ConsoleAttach.cs` + empirical testing.

**Compatibility with pwsh-direct workflow:** Option B requires no change to the
`tiny11maker.ps1` direct-invocation path. The `--log` flag is consumed by `HeadlessRunner`
before passing remaining args to `tiny11maker.ps1`. Pwsh-direct users already use
`Start-Transcript` or `| Tee-Object` natively against their own pwsh console and are
unaffected.

## Findings

### WARNING-1: Pipe redirection silently drops all child output

**Location:** `launcher/Headless/HeadlessRunner.cs:35` (AttachConsole call) + lines 58-59
(RedirectStandardOutput = false).

**Description:** When the EXE is invoked with `| Tee-Object` or `> out.log` from pwsh, the
parent process has no console. `AttachConsole(-1)` returns FALSE; the error is ignored.
`Console.Out` is never refreshed. The child `powershell.exe` is then spawned inheriting the
EXE's null stdout handle. All child output goes to null. `out.log` is empty. This is the
user's P1 smoke testing blocker.

**Severity:** WARNING (UX gap that significantly hampers troubleshooting).

---

### WARNING-2: `Console.SetOut` not refreshed after `AttachConsole`

**Location:** `launcher/Interop/ConsoleAttach.cs:17-23`

**Description:** On .NET, `Console.Out` is lazily initialized once at process start. Calling
`AttachConsole` does not automatically update the managed `Console.Out` / `Console.Error`
writers to use the newly attached console handle. Even when `AttachConsole` succeeds (no
pipe, normal terminal), the `Console.Error.WriteLine(...)` calls in `HeadlessRunner.cs:46`
and `HeadlessRunner.cs:73` may go to the old (null) handle rather than the attached console.
Empirically — if the terminal is Windows Terminal or cmd.exe and the EXE's initial stdout
handle from process creation is valid, it _might_ work; but it is not guaranteed by the
.NET/Win32 contract. Resolving this requires `Console.SetOut` + `Console.OpenStandardOutput()`
after a successful `AttachConsole`.

**Severity:** WARNING

---

### INFO-1: `AttachConsole` return value silently ignored

**Location:** `launcher/Interop/ConsoleAttach.cs:22`

**Description:** `AttachConsole(ATTACH_PARENT_PROCESS)` return value is discarded with
comment "Best-effort ... Failures are non-fatal." For the headless path this is too
permissive — a failed attach means all output is dropped, which is observable and
user-visible. The method should at minimum return the `bool` result so callers can
decide whether to fall back to `AllocConsole` (for interactive use) or switch to file
logging (for piped/CI use).

**Severity:** INFO

---

### INFO-2: Child spawned with `powershell.exe` (PS 5.1), not `pwsh.exe`

**Location:** `launcher/Headless/HeadlessRunner.cs:54`

**Description:** `FileName = "powershell.exe"` — this is correct per the pwsh-from-pwsh
block requirement (tiny11maker.ps1:93-114), but it relies on `powershell.exe` being on
system PATH. Per the Local-Dependencies-Only architecture rule this is a pre-existing known
exception (Windows built-in, not a vendored dep); flag here for completeness.

**Severity:** INFO (pre-existing, not introduced by this audit)

---

### INFO-3: `FreeConsole` called in `finally` — may corrupt parent prompt

**Location:** `launcher/Headless/HeadlessRunner.cs:85`, `launcher/Interop/ConsoleAttach.cs:25`

**Description:** `FreeConsole()` is called after the child process exits. On some Windows
Terminal / conhost combinations, a WinExe process that attached to the parent console and
then calls `FreeConsole` before the parent reads back its prompt can cause the terminal to
re-display the prompt prematurely (cursor appears at wrong position). This is a cosmetic
issue in interactive terminal use; it is a non-issue in the piped/CI case. Known limitation
of the attach-then-free pattern. Low risk; document and leave unless user reports it.

**Severity:** INFO

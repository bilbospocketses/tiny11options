## Findings

### BLOCKER

---

#### BLOCKER-1: `tiny11maker-from-config.ps1` missing from `StaticResources` — headless mode cannot run config-based builds

**File:** `launcher/Headless/HeadlessRunner.cs` line 15-31  
**Cross-ref:** `launcher/tiny11options.Launcher.csproj` lines 81-82

`tiny11maker-from-config.ps1` and `tiny11Coremaker-from-config.ps1` are both embedded as `<EmbeddedResource>` in the csproj. Neither appears in `StaticResources[]` in `HeadlessRunner.cs`.

The breadcrumb says "headless only invokes `tiny11maker.ps1`" and that `tiny11Coremaker-from-config.ps1` is intentionally absent. After reading the code:

- `HeadlessRunner.cs` only spawns `tiny11maker.ps1` (line 50: `var ps1Path = Path.Combine(tempDir, "tiny11maker.ps1")`). It does NOT support config-file mode or core mode from the CLI.
- `tiny11maker.ps1` imports its own module chain directly via `$PSScriptRoot` relative paths (verified via grep). It does NOT dot-source or call `tiny11maker-from-config.ps1`.

So `tiny11Coremaker-from-config.ps1` absence from `StaticResources` is CONFIRMED intentional.

However, `tiny11maker-from-config.ps1` is more ambiguous. Reading `tiny11maker.ps1` source (verified in grep above), `tiny11maker.ps1` does NOT call `tiny11maker-from-config.ps1` internally, so it is also not required at headless runtime.

**But the concern is different:** `tiny11maker-from-config.ps1` is embedded but never extracted in headless mode. If a future CLI flag routes to config-based builds, this script silently won't be on disk. At present, headless mode has no `-ConfigPath` or coreMode dispatch — `HeadlessRunner.BuildPwshArgLine` (line 89-100) passes user args verbatim to `tiny11maker.ps1`, so there is no current runtime failure.

**Revised severity:** WARNING (not BLOCKER). The absence does not cause a present-day runtime failure. Reclassified below.

#### BLOCKER-1 (REAL): Version mismatch — `app.manifest` says `1.0.1.0`, assembly attributes say `1.0.0.0`

**File:** `launcher/app.manifest` line 3: `version="1.0.1.0"`  
**File:** `launcher/obj/Release/.../tiny11options.Launcher.AssemblyInfo.cs` lines 15, 19: `AssemblyFileVersion("1.0.0.0")`, `AssemblyVersion("1.0.0.0")`  
**File:** `launcher/obj/Debug/.../tiny11options.Launcher.AssemblyInfo.cs` lines 15, 19: same `1.0.0.0`

The `app.manifest` `assemblyIdentity version` was bumped to `1.0.1.0`, but the `<Version>`, `<AssemblyVersion>`, or `<FileVersion>` properties are entirely absent from `launcher/tiny11options.Launcher.csproj`. The MSBuild SDK therefore defaults the assembly to `1.0.0.0`.

This creates a mismatch:
- The Windows manifest identity reports `1.0.1.0`
- The .NET assembly version is `1.0.0.0`
- `Assembly.GetExecutingAssembly().GetName().Version` returns `1.0.0.0`
- `MainWindow.ResolveUiCacheDir()` (line 203) and `MainWindow.ResolveResourcesDir()` (line 212) both key their cache directory path on `Assembly.GetExecutingAssembly().GetName().Version?.ToString()` — so the GUI caches to `...\ui-cache\1.0.0.0\` and `...\resources-cache\1.0.0.0\` instead of `...\1.0.1.0\`

**Impact:** A user upgrading from v1.0.0 to v1.0.1 will have the same cache directory path as v1.0.0. The hash-based `ExtractIfManifestChanged` mechanism will detect content changes (SHA256 will differ since `Tiny11.PostBoot.psm1` is new) and re-extract correctly. So the UI/resources cache will still update. However:
- Velopack delta update logic may be confused if the assembly version doesn't advance. Velopack reads `AssemblyInformationalVersion` (git hash) and the Velopack package version for update checks — but Windows Installer/Add-Remove-Programs version surfaces `FileVersion`, which stays `1.0.0.0`.
- The CHANGELOG entry for `[1.0.1]` (line 16) and README section "Post-boot cleanup task (v1.0.1+)" (line 48) document this as a versioned release, inconsistent with the artifact's `1.0.0.0` assembly version.

**Severity: BLOCKER.** The csproj is missing a `<Version>1.0.1.0</Version>` (or `<AssemblyVersion>`/`<FileVersion>`) entry. Without it the shipped EXE self-reports as `1.0.0.0`, defeating the version-gated UI cache dirs, confusing any tooling that reads file version (Windows Explorer Properties, Velopack release manifests, installer logs).

#### BLOCKER-2: `HeadlessRunner` has no CLI flag for `--no-post-boot-cleanup` — headless post-boot cannot be opted out

**File:** `launcher/Headless/HeadlessRunner.cs` lines 89-100 (`BuildPwshArgLine`)

`HeadlessRunner.BuildPwshArgLine` passes `userArgs` verbatim to `tiny11maker.ps1`. The user could theoretically pass `-NoPostBootCleanup` as a raw CLI arg and it would be forwarded. However:

1. There is no documented CLI surface in `HeadlessRunner` for controlling the post-boot-cleanup flag. `Program.cs` line 16 routes to `HeadlessRunner.Run(args)` for any headless invocation, but nowhere does `HeadlessRunner` parse or validate args.
2. The CHANGELOG (line 28) states: "C# launcher (`BuildHandlers.cs`) reads `installPostBootCleanup` from the start-build payload (default true when missing) and appends `-NoPostBootCleanup` to the wrapper invocation when JS sends `installPostBootCleanup: false`." This is ONLY for GUI mode. Headless mode has no equivalent.
3. The headless user's only path is to pass `-NoPostBootCleanup` as a raw CLI string, which is fragile and undocumented.

**Severity: BLOCKER** — headless invocations always default to installing the post-boot cleanup task with no documented way to opt out via the launcher CLI. (The underlying `tiny11maker.ps1` does accept `-NoPostBootCleanup`; the problem is the launcher neither documents nor exposes this.)

#### WARNING-1: `tiny11-iso-validate.ps1` and `tiny11-profile-validate.ps1` embedded but not in `StaticResources`

**File:** `launcher/tiny11options.Launcher.csproj` lines 83-84  
**File:** `launcher/Headless/HeadlessRunner.cs` lines 15-31

These two scripts are embedded as `<EmbeddedResource>` but absent from `StaticResources[]`. Whether they're needed in headless mode depends on whether `tiny11maker.ps1` dot-sources them during a CLI build. The current file list and `HeadlessRunner` comments suggest these are GUI-side validation scripts (called from `IsoHandlers.cs` / `ProfileHandlers.cs`) not needed for the build pipeline.

**UNCERTAIN:** A full read of `tiny11maker.ps1`'s import/dot-source chain would confirm. If either script is called transitively from the main pipeline, headless builds would silently skip validation. The `EmbeddedResources.ExtractTo` method throws `FileNotFoundException` if a resource is requested but not found (line 20-24), so the failure would be loud — but only if the script is actually requested by name. If the script is never requested, absence is silent.

**Severity: WARNING** — low probability given the naming convention (`validate` prefix = standalone validators), but should be confirmed.

#### WARNING-2: `tiny11maker-from-config.ps1` and `tiny11Coremaker-from-config.ps1` embedded but not extracted in headless mode — silent API gap for future CLI expansion

**File:** `launcher/tiny11options.Launcher.csproj` lines 81-82  
**File:** `launcher/Headless/HeadlessRunner.cs` lines 15-31

Both config-wrapper scripts are embedded but not in `StaticResources`. This is intentional today (headless only invokes `tiny11maker.ps1`). However, if a future CLI addition routes headless to config-based builds (the GUI already does this for standard builds at `BuildHandlers.cs` line 141-143), these scripts would silently not be on disk at runtime. There is no compile-time or test-time assertion that `StaticResources` stays consistent with "what headless actually needs."

**Severity: WARNING** — no current breakage, but a risky pattern as the headless API expands.

#### WARNING-3: `BuildHandlersTests.cs` — no test for default-true behavior when `installPostBootCleanup` is absent from payload

**File:** `launcher/Tests/BuildHandlersTests.cs`

The four PostBoot tests (lines 413-451) test:
- `BuildStandardArgs` with explicit `installPostBootCleanup: true` — `-NoPostBootCleanup` absent (line 414-421)
- `BuildStandardArgs` with explicit `installPostBootCleanup: false` — `-NoPostBootCleanup` present (line 423-431)
- `BuildCoreArgs` with explicit `installPostBootCleanup: true` — absent (line 433-441)
- `BuildCoreArgs` with explicit `installPostBootCleanup: false` — present (line 443-451)

**Missing test:** No test calls `HandleAsync` with a payload that omits `installPostBootCleanup` entirely and asserts the default-true behavior (i.e., that `payload?["installPostBootCleanup"]?.GetValue<bool>() ?? true` yields `true` when the field is absent). The default is at `BuildHandlers.cs` line 114. If that `?? true` default were accidentally changed to `?? false`, no test would catch it — the four existing tests pass explicit values and skip the deserialization-default path.

This gap matters because the JS-side default is also `true` (`state.installPostBootCleanup: true` at `ui/app.js` line 80), so the defaults must agree. A JS change that omits the field from the payload when `true` (as an optimization) would silently invert behavior if the C# default flipped.

**Severity: WARNING** — no current bug, but a coverage gap on the most important default path.

#### INFO-1: `src/Tiny11.Core.psm1` embedded but intentionally absent from `StaticResources` — no test asserts this

**File:** `launcher/tiny11options.Launcher.csproj` line 76  
**File:** `launcher/Headless/HeadlessRunner.cs` lines 15-31

The breadcrumb is correct: `Tiny11.Core.psm1` is embedded but absent from `StaticResources` because headless calls `tiny11maker.ps1` (Worker path) not the Core path. This is verified — there is no Core import in `tiny11maker.ps1`'s chain per the grep results. Absence is intentional.

#### INFO-2: GUI mode resource extraction is NOT a hardcoded list — it is manifest-driven; no parallel `GuiResources[]` gap exists

**File:** `launcher/MainWindow.xaml.cs` lines 221-291

The audit spec asked whether GUI mode has an analogous hardcoded `GuiResources[]` that could miss PostBoot. It does not. GUI mode uses `ExtractRuntimeResourcesIfNeeded` (line 228-232), which calls `ExtractIfManifestChanged` with a filter predicate: extract everything that is NOT `ui/` and NOT `test-fixture.txt`. This is fully dynamic — it walks `Assembly.GetManifestResourceNames()` and extracts all matching resources. Adding a new `<EmbeddedResource>` to the csproj automatically makes it available in GUI mode on next launch; no parallel list to update.

The CHANGELOG note (line 53) that "GUI path unaffected" after the StaticResources regression is confirmed correct.

---

#### INFO-3: `EmbeddedResourcesTests.cs` `RealResource_IsEmbedded` does NOT cover `src/Tiny11.PostBoot.psm1`

**File:** `launcher/Tests/EmbeddedResourcesTests.cs` lines 44-55

The `RealResource_IsEmbedded` theory test covers `tiny11maker.ps1`, `autounattend.template.xml`, `src/Tiny11.Iso.psm1`, `ui/index.html`. It does not include `src/Tiny11.PostBoot.psm1`. Adding an `[InlineData("src/Tiny11.PostBoot.psm1")]` case would give a compile-time assert that PostBoot is embedded, which would have caught the original b35d73a-class regression at test-run time rather than smoke time. Low-cost addition.

#### INFO-4: Stale-file hazard across version upgrades is mitigated by hash-based extraction, but temp-dir headless runs are NOT protected

**File:** `launcher/MainWindow.xaml.cs` lines 244-291 (GUI hash-based extraction)  
**File:** `launcher/Headless/HeadlessRunner.cs` lines 110-148 (headless temp dir)

GUI mode: `ExtractIfManifestChanged` computes a SHA256 over all resource names + bytes and compares to a `.extracted` marker file in the versioned cache dir. If content changed (e.g., v1.0.0 -> v1.0.1 adds PostBoot), the cache dir is wiped and re-extracted. Stale-file hazard is fully addressed.

Headless mode: uses a per-process-ID temp dir (`tiny11options-<pid>`), so each headless invocation gets a fresh directory. No stale-file hazard possible by construction. `TryCleanup` (line 132-136) deletes the dir on exit.

The audit spec concern (Item 8) is not a real hazard in practice. Noted here for completeness.

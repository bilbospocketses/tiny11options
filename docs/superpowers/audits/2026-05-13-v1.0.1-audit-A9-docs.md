## Findings

### BLOCKER

#### B1 — Version bump only touches `app.manifest`, not `.csproj`

**Claim:** Plan Task 28 says to bump `<Version>1.0.1</Version>` in `launcher/tiny11options.Launcher.csproj`.

**Code reality:** Commit `7a7d6a2` only modifies `launcher/app.manifest` (1 file changed per `git show 7a7d6a2 --stat`). The `.csproj` has NO `<Version>` element — only a `<PackageReference Include="Velopack" Version="0.0.1298" />`. The `app.manifest` correctly reads `version="1.0.1.0"`.

**Impact:** The Velopack release pipeline reads the version from the `.csproj` `<Version>` property (per Velopack docs) to label the release artifact. If `.csproj` has no `<Version>` tag, the build will default to `1.0.0.0` (the MSBuild default), NOT `1.0.1`. The `app.manifest` version is only the Win32 assembly version (shown in file properties); it does NOT drive the Velopack update channel version comparison. Users already on v1.0.0 will NOT receive an update offer for this release because Velopack will see the package as `1.0.0`, not `1.0.1`.

**Severity:** BLOCKER — will silently break the auto-update flow.

**Fix:** Add `<Version>1.0.1</Version>` inside the `<PropertyGroup>` in `launcher/tiny11options.Launcher.csproj`.

#### B2 — `[Unreleased] ### Fixed` contains an item that is explicitly NOT fixed

**Claim:** CHANGELOG.md `[Unreleased] ### Fixed` line 14: "NOT FIXED in v1.0.1; tracked for v1.0.2 follow-up" (WinExe-stdout `ConsoleAttach` limitation).

**Code reality:** The item is not fixed. Placing it under `### Fixed` in a CHANGELOG section is incorrect and will mislead users reading the changelog to understand what was fixed. Users reading the `[Unreleased]` Fixed section will infer this item was addressed.

**Severity:** BLOCKER (misleading to users reading changelog).

**Fix:** Move the WinExe-stdout item out of `### Fixed` and into a new `### Known Issues` or `### Documented` subsection within `[Unreleased]`. Example:

```markdown
### Known Issues (documented 2026-05-13 -- discovered via P1 smoke)
- **`tiny11options.exe` stdout-to-pipe limitation** (NOT FIXED in v1.0.1; tracked for v1.0.2). ...
```

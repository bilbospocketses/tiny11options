## Findings

### BLOCKER

#### [B1] Core.psm1 top-level `-Force -Global` import of PostBoot cascades into Actions.Registry's no-Global Hives re-load

- **File:** `src/Tiny11.Core.psm1:14`
- **Description:** `Import-Module Tiny11.PostBoot.psm1 -Force -Global` at module-body level triggers PostBoot's module body, which runs `Import-Module Tiny11.Actions.psm1 -Global` (no -Force — the Regression 2 fix). Since PostBoot is being force-loaded, its body executes fresh. The `-Global` on PostBoot's Actions import causes Actions' body to execute at global scope. Actions then runs `Import-Module Tiny11.Actions.Registry.psm1 -Force -Global`. Registry's body executes `Import-Module Tiny11.Hives.psm1 -Force` with **no -Global**. This re-loads Hives at the scope of the Actions.Registry module context, not global. The Core module's top-level load of PostBoot thus re-enacts the Regression 2 mechanism via the force-cascade: the global Hives that Worker.psm1 established (if Worker was loaded first) gets **shadowed by a non-global reload**. The key distinction from Regression 2 is that `-Force` is on Core's PostBoot import (line 14), not on PostBoot's Actions import (which is now no-Force). But `-Force` on the outer import still forces PostBoot's body to re-execute, and PostBoot's body still triggers Actions' body (even without -Force on that call, because PostBoot is a fresh module that hasn't loaded Actions yet), which triggers Registry's body, which force-reloads Hives without -Global.
- **When it bites:** The Core mode pipeline. When `Invoke-Tiny11CoreBuildPipeline` eventually calls `Mount-Tiny11AllHives`, it resolves the command in the calling scope. If Hives was demoted from global by the Core module-body cascade, the function is missing.
- **UNCERTAIN:** Whether the no-Force on PostBoot's Actions import (`src/Tiny11.PostBoot.psm1:9`) actually suppresses the cascade when PostBoot itself is force-loaded. If PostBoot's module state is cleared by the outer -Force, PowerShell may execute PostBoot's body treating it as a fresh session, which would make the no-Force on the inner Actions import equivalent to a first-time load — meaning Actions WILL execute (not skipped). If that's true, B1 is a confirmed blocker. **Experiment to confirm:** load Worker.psm1 (establishes global Hives), then `Import-Module Tiny11.Core.psm1 -Force`, then call `Mount-Tiny11AllHives` in the global scope and check `(Get-Module Tiny11.Hives).ExportedFunctions`.
- **Recommended action:** Add `-Global` to `src/Tiny11.Actions.Registry.psm1:2`'s Hives import. This is the same structural fix as Regression 2 identified as the missing complement. Alternatively: audit whether Core.psm1:14 can drop `-Force`; if Core is always the first loader of PostBoot in its session, -Force is unnecessary and dropping it would prevent the cascade.

#### [B2] `Invoke-Tiny11CoreBuildPipeline` force-reloads Hives mid-pipeline without `-Global`

- **File:** `src/Tiny11.Core.psm1:1095`
- **Description:** Inside the body of `Invoke-Tiny11CoreBuildPipeline` (a function, not module body), line 1095 runs `Import-Module Tiny11.Hives.psm1 -Force` with no `-Global`. In PowerShell, `Import-Module` inside a function body without `-Global` places the module into the local (function) scope, not the global session scope. This does not propagate `Mount-Tiny11Hive`, `Dismount-Tiny11Hive`, etc. to the global scope. The pipeline then calls `Mount-Tiny11Hive` at lines 1268 and 1401. Those calls resolve against the scope chain at the call site. If the global Hives (loaded by Core module body via the PostBoot cascade or by a prior Worker load) is still present, the calls succeed — but the function-scoped reload is redundant and potentially dangerous: if the global scope copy is somehow absent (e.g. if Core module is loaded standalone for tests), the function-scoped import will satisfy the immediate call, but only for the duration of the function's scope. Subsequent calls from outside this function would still see missing commands.
- **Why it matters:** The defensive intent (ensuring Hives is loaded before the pipeline runs) is correct, but the implementation is wrong. The force reload without -Global does not achieve the stated goal. It creates a false sense of safety while leaving a scope gap.
- **Recommended action:** Change `src/Tiny11.Core.psm1:1094-1095` to `Import-Module ... -Force -Global` to match Worker.psm1's established pattern, OR remove these defensive imports if Core's module-body load of PostBoot (and its cascade) reliably loads Hives globally. Do not leave -Force without -Global on a module that downstream code calls by name from outside the function.

#### [B3] `Invoke-Tiny11CoreBuildPipeline` force-reloads Worker mid-pipeline without `-Global`

- **File:** `src/Tiny11.Core.psm1:1452`
- **Description:** `Import-Module Tiny11.Worker.psm1 -Force` at line 1452 — inside a function body, no `-Global`. Worker.psm1's module body contains five `-Force -Global` imports (Actions, Hives, Iso, Autounattend, PostBoot). When Worker is loaded inside a function without -Global, PowerShell executes Worker's module body. The sub-imports in Worker's body carry `-Force -Global`, so they ARE placed globally. However, Worker itself is in the function scope. The specific concern: `Resolve-Tiny11Oscdimg` is called at line 1455 on the very next line — this works because the function is in the (function-scoped) Worker module. But the broader pattern is fragile: Worker's module-body `-Force` on all its sub-imports now fires AGAIN (Force means re-execute body), which re-runs the PostBoot cascade (PostBoot -> Actions -> Actions.Registry -> Hives without -Global) from inside a function scope.
- **Chain:** Core:1452 `-Force` on Worker -> Worker body runs -> Worker:6 `-Force -Global` on PostBoot -> PostBoot body runs (fresh module, so body executes regardless of -Force absence on Actions import) -> PostBoot:9 `-Global` on Actions (no -Force, but first run of this PostBoot instance so Actions IS loaded) -> Actions:2 `-Force -Global` on Registry -> Registry:2 `-Force` on Hives (no -Global) -> Hives demoted from global scope.
- **Why it matters:** Any call to Hives functions after line 1452 in `Invoke-Tiny11CoreBuildPipeline` (there are none directly, but subsequent pipeline phases called from here use Hives) risks `CommandNotFound` if the global Hives was demoted.
- **UNCERTAIN:** Whether the Worker's `-Force -Global` on PostBoot causes PostBoot's body to run with PostBoot installed at global scope, which would then mean PostBoot's own module-body Actions import fires at global scope and produces a fresh cascade. The critical question is scope inheritance when a `-Force -Global`-loaded module's body calls `Import-Module` — does that nested call inherit global scope as its "caller"? If yes, the cascade is global and the Hives demotion is real. **Experiment:** add `Write-Host "PostBoot body executing, caller: $MyInvocation.MyCommand"` to PostBoot's module body and observe when Core:1452 fires.
- **Recommended action:** Add `-Global` to `src/Tiny11.Core.psm1:1452`. Additionally, remove the redundant PostBoot/Worker re-imports in Core's pipeline function if all required modules are already established by Core's module-body import at line 14. The current defensive pattern of mid-pipeline imports creates the scope hazard they're trying to prevent.

### WARNING

#### [W1] Defensive `Import-Module Tiny11.Iso.psm1 -Force` inside pipeline function (two occurrences)

- **File:** `src/Tiny11.Core.psm1:1094` and `src/Tiny11.Core.psm1:1473`
- **Description:** `Tiny11.Iso.psm1` is a leaf module (no cascading imports). The `-Force` without `-Global` inside a function body reloads it at function scope. `Tiny11.Iso.psm1` is not imported at Core's module top level (only PostBoot is, at line 14). Worker loads Iso globally (Worker:4). If Core pipeline runs in the same session as Worker (the normal case), the global Iso from Worker remains. If Core runs standalone (tests, Core-mode headless), the function-scope reload is the only load — which means Iso functions like `Mount-Tiny11Source` and `Dismount-DiskImage` are only available for the duration of `Invoke-Tiny11CoreBuildPipeline`. This works for the pipeline function itself, but any future caller outside this function that expects Iso to be globally visible after calling the pipeline will be surprised.
- **Why it matters:** Lower-risk than B2/B3 because Iso is a leaf and calls are within the same function. But the pattern diverges from Worker's established -Force -Global discipline.
- **Recommended action:** Change to `-Force -Global` on both Iso imports, OR hoist Iso to Core's module-body import block alongside the PostBoot import.

#### [W2] Worker.psm1 and Core.psm1 both load PostBoot globally at module body — double-load risk

- **File:** `src/Tiny11.Worker.psm1:6` and `src/Tiny11.Core.psm1:14`
- **Description:** Both Worker and Core import PostBoot with `-Force -Global` at their respective module bodies. In the GUI path, the launcher loads Core.psm1 (or a wrapper that loads Core), which at module body fires the PostBoot cascade. If Worker is subsequently loaded (or vice versa), a second `-Force -Global` on PostBoot fires PostBoot's body again. Since PostBoot's Actions import is no-Force, and Actions is already globally loaded, the no-Force import is a no-op — which means Actions.Registry's Hives import is NOT re-run. So the double-load is currently benign. However, this property depends on PostBoot's Actions import remaining no-Force. Any future edit that adds `-Force` back to PostBoot:9 would reactivate the full Regression 2 mechanism from both loader paths simultaneously.
- **Recommended action:** Document the dependency explicitly (PostBoot MUST NOT use -Force on Actions) with a comment cross-referencing both loading sites. Consider whether Core needs to import PostBoot directly at module body at all, vs. relying on Worker's top-level import when both are in the same session.

#### [W3] Five PostBoot test files load PostBoot in isolation without prior Actions load

- **Files:** `tests/Tiny11.PostBoot.FormatArgs.Tests.ps1:5`, `tests/Tiny11.PostBoot.HeaderBlock.Tests.ps1:5`, `tests/Tiny11.PostBoot.Helpers.Golden.Tests.ps1:5`, `tests/Tiny11.PostBoot.TaskXml.Tests.ps1:5`, `tests/Tiny11.PostBoot.SetupComplete.Tests.ps1:5`
- **Description:** These five test files do `Import-Module PostBoot -Force -DisableNameChecking` as the only import, with no prior Actions load. PostBoot's body runs `Import-Module Actions -Global` (no -Force). Since Actions is not yet loaded, this IS a first-time load and succeeds — but Actions itself imports Registry (-Force -Global), which imports Hives (-Force, no -Global). In a Pester BeforeAll block, the scope of `Import-Module` without -Global may not be the same global scope that subsequent `It` blocks resolve against. Whether `Get-Tiny11ActionOnlineCommand` (an Actions export) is visible inside `It` blocks depends on whether the BeforeAll's global import actually went global. The tests currently only test `Format-PSNamedParams`, `New-Tiny11PostBootTaskXml`, `New-Tiny11PostBootSetupCompleteScript`, and `$script:headerBlock` / `$script:helpersBlock` content — functions that are defined directly in PostBoot and do not call through to Actions. So tests pass. But the test load does install Actions/Hives at whatever scope PostBoot's body ends up at, meaning: **if any function in these tests were changed to call `Get-Tiny11ActionOnlineCommand`, the test would fail under some Pester runspace configurations even though the production path works fine.**
- **Recommended action:** Add `Import-Module Tiny11.Actions.psm1 -Force -DisableNameChecking` before the PostBoot import in all five files, to match the ordering in Generator.Tests.ps1 and Install.Tests.ps1. This makes the test load order match production and future-proofs against test additions.

### INFO

#### [I1] `Actions.Registry.psm1` Hives import is -Force without -Global — the persistent structural root cause

- **File:** `src/Tiny11.Actions.Registry.psm1:2`
- **Description:** `Import-Module Tiny11.Hives.psm1 -Force -DisableNameChecking` (no -Global). This is the structural root cause of Regression 2 and the active risk factor for B1 and B3. The reason -Global was not added here when fixing Regression 2 was that the fix was applied one layer up (removing -Force from PostBoot's Actions import) rather than adding -Global here. Both fixes address the same symptom, but fixing it at the Actions.Registry layer would be the more robust approach: it would make Actions.Registry's Hives load unconditionally global regardless of what caller-scope chain led there.
- **Why it is INFO rather than BLOCKER now:** The Regression 2 fix (no -Force on PostBoot's Actions import) means the cascade that would hit this path is currently blocked at PostBoot:9. B1 and B3 describe scenarios where the block can be bypassed. This item captures the underlying structural gap.
- **Recommended action:** Add `-Global` to `src/Tiny11.Actions.Registry.psm1:2`. This is the defensive fix that makes the scope chain resilient regardless of who calls Actions.Registry and from what scope. It is low-risk since Hives is a pure-leaf utility module.

## Verification notes

- All findings derived from direct source reads. No inference from commit messages or spec documents.
- The most uncertain finding is B1/B3 — whether PowerShell considers a no-Force `Import-Module` inside a freshly force-loaded module's body as "already loaded" or "first time." The behavioral question is: after `Import-Module PostBoot -Force` clears PostBoot's module state, does the subsequent `Import-Module Actions -Global` (inside PostBoot's body, no -Force) treat Actions as "already loaded in global scope" (no-op) or "not loaded in this module's context" (executes body)? If Actions is already in `Get-Module` at global scope, PowerShell's default behavior is to skip re-execution when -Force is absent, even in a fresh module context. This means the B1/B3 cascade risk hinges on whether the outer session already had Actions loaded globally before Core/Worker's force reload of PostBoot. In a clean session (Core mode standalone), Actions would NOT be pre-loaded, making the cascade active.
- The recommended experiment to resolve B1/B3 uncertainty: In a fresh PowerShell session, run `Import-Module src/Tiny11.Core.psm1 -Force -Global`, then `Get-Module Tiny11.Hives | Select-Object Name, Scope` and observe whether scope is `Global` or `Local`.
- `Tiny11.Bridge.psm1`, `Tiny11.WebView2.psm1`, `Tiny11.Catalog.psm1`, `Tiny11.Selections.psm1`, `Tiny11.Autounattend.psm1`, and `Tiny11.Iso.psm1` contain no `Import-Module` calls; they are clean leaf modules with no cascade risk.

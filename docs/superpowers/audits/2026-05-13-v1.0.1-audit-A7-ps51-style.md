## Blocker

### BLOCKER — `$args` is a PS automatic variable; assigning to it is undefined behavior

**File:** `src/Tiny11.Core.psm1`, line 499  
**Function:** `Invoke-CoreIcacls`

```powershell
$args = @($Path, '/grant', 'Administrators:F')
if ($Recurse) { $args += @('/T', '/C') }
Start-CoreProcess -FileName 'icacls.exe' -Arguments $args
```

`$args` is a PowerShell automatic variable — it holds the unbound positional arguments passed to a function or script. Assigning to it is not blocked by the parser, but the behavior is version-dependent and unreliable. In PS 5.1, the assignment usually works within the function scope (PS creates a local shadow), but it is fragile: `Set-StrictMode -Version Latest` is in effect at the module level, and the interaction with `$args` as both the automatic variable and a local variable is implementation-defined.

**PS 7 behavior also varies:** the assignment compiles but may produce confusing side effects when strict-mode is active.

**Fix:** rename to `$icaclsArgs` (matching the `$takeownArgs` name used in the parallel `Invoke-CoreTakeown` function at lines 486–488).

This is a **BLOCKER** because the function is called in several critical paths (WinSxS wipe Phase 8, filesystem-removal Phase 7, install.wim + boot.wim prep in the Core pipeline) and a silent misfire here would silently fail to grant Administrators:F, causing subsequent `Remove-Item` to fail with access-denied.

### WARNING 4 — `Write-Marker` in wrapper scripts uses non-advanced function syntax

**File:** `tiny11maker-from-config.ps1` line 37; `tiny11Coremaker-from-config.ps1` line 38

```powershell
function Write-Marker($Type, [hashtable]$Payload) {
```

Non-advanced function (no `[CmdletBinding()]`). Not flagged as a style violation because `Write-Marker` is a private helper with no callers outside the script, but the `[hashtable]` type constraint in the positional-style `param` list is unusual — positional param constraints like this work in 5.1 but could confuse maintainers. Using a proper `param()` block with named params would be cleaner.

### WARNING 5 — `Install-Tiny11PostBootCleanup`: `$Enabled` param uses bare `[bool]` default instead of `[switch]`

**File:** `src/Tiny11.PostBoot.psm1`, line 314

```powershell
[bool]  $Enabled = $true
```

The pattern is used consistently across Worker and Core wrappers (`-InstallPostBootCleanup $true/([bool]...)`) and works correctly. However, a `[bool]` parameter with a default is slightly awkward to pass as `-Enabled:$false` vs `-Enabled $false` (both work in 5.1, but the colon form is needed when passing `$false`). Callers all use the explicit `$true`/`$false` form, so this is a soft warning about API ergonomics, not a bug.

### WARNING 6 — `Format-PSNamedParams` not exported from `Tiny11.PostBoot.psm1` — or rather, it is exported but intended as internal

**File:** `src/Tiny11.PostBoot.psm1`, line 346

`Format-PSNamedParams` is exported in the `Export-ModuleMember` list. It is a formatting helper used internally by `New-Tiny11PostBootCleanupScript`. Exporting it pollutes the module surface with a helper that has no use-case outside the module. Consider removing it from the export list, or renaming it with a private prefix convention (e.g. `ConvertTo-Tiny11PSNamedParams`).

### WARNING 7 — `tiny11maker.ps1`: `$InstallPostBootCleanup = $true` default param alongside `[switch]$NoPostBootCleanup`

**File:** `tiny11maker.ps1`, lines 46–48

```powershell
[bool]$InstallPostBootCleanup = $true,
[switch]$NoPostBootCleanup,
```

Two params covering the same feature toggle. The downstream expression `([bool]($InstallPostBootCleanup -and -not $NoPostBootCleanup))` correctly handles both. But the dual-param pattern appears in all three entry-point scripts and creates a minor API inconsistency: `tiny11maker.ps1` and both `*-from-config.ps1` wrappers all carry both params. If `$NoPostBootCleanup` is meant to be the user-facing switch, `$InstallPostBootCleanup` should be demoted to an internal default; if both are for different callers (PowerShell CLI vs. JSON payload), that should be documented.

### WARNING 8 — `HKU:` PSDrive not guarded in generated tiny11-cleanup.ps1

**File:** `src/Tiny11.PostBoot.psm1`, `$script:helpersBlock` (lines 71–103)

`Set-RegistryValueForAllUsers` and `Remove-RegistryKeyForAllUsers` both assume `HKU:\` is a mounted PSDrive. The SYSTEM context under which `tiny11-cleanup.ps1` runs (scheduled task, every boot, daily, on WU event) does NOT automatically have `HKU:` as a PS drive. On PS 5.1, `Get-ChildItem 'HKU:\'` will throw `Drive 'HKU' does not exist.` unless the caller first runs:

```powershell
if (-not (Get-PSDrive 'HKU' -ErrorAction SilentlyContinue)) {
    New-PSDrive -Name HKU -PSProvider Registry -Root HKEY_USERS | Out-Null
}
```

The same pattern is presumably needed in the generated `tiny11-wu-enforce.ps1`; that script does NOT use `HKU:\` (it uses `HKLM:\` throughout), so it is clean. But `tiny11-cleanup.ps1`'s per-user fan-out functions will silently fail for any NTUSER-hive registry action unless `HKU:` is mapped. Severity: WARNING because the guards should be in the `$script:headerBlock` or the fan-out functions themselves, not left to ambient context.

**Files:** `src/Tiny11.Core.psm1` (lines 45, 113, 128, 877); `src/Tiny11.PostBoot.psm1` (lines 234, 286)

Several zero-param data-accessor functions use the compact `[CmdletBinding()] param()` form — e.g. `New-Tiny11PostBootTaskXml` at line 234, `New-Tiny11PostBootSetupCompleteScript` at line 286. Others (`Get-Tiny11CoreAppxPrefixes`, `Get-Tiny11CoreFilesystemTargets`, `Get-Tiny11CoreScheduledTaskTargets`, `Get-Tiny11CoreWuEnforceTaskXml`) lack `[CmdletBinding()]` entirely. Subjective — zero-param functions that just return data don't need common parameters — but if the convention is to be `[CmdletBinding()]`-everywhere, these are the gaps.

### INFO 2 — Verb compliance: `Format-PSNamedParams` uses `Format` verb

`Format-*` is a standard PS verb (Get-Verb returns it), so no PSScriptAnalyzer violation. But the function actually _renders_ a string, not formats output for display. `ConvertTo-Tiny11PSArgs` would be slightly more semantically correct. Subjective.

### INFO 3 — String quoting is consistent and correct throughout

All interpolated strings correctly use `"..."` and literals use `'...'`. No misuses found in any of the scoped files. Clean.

### INFO 4 — `#region`/`#endregion` not used

None of the files use `#region` / `#endregion`. This is a style call; the files are short enough (PostBoot, Actions.* modules) or internally commented with `# ---` section headers (Core, Worker) that region folding isn't strictly needed.

## Findings

### BLOCKER

#### B1 — `tiny11maker.ps1` self-elevation relaunch silently drops `$InstallPostBootCleanup = $false` via mistyped serialization

**File:** `tiny11maker.ps1:59-74` (`Build-RelaunchArgs`) + `tiny11maker.ps1:118` (UAC relaunch path)

`Build-RelaunchArgs` iterates `$PSBoundParameters`. When the param is typed `[bool]` (not `[switch]`), the `else` branch fires (line 68-71) and serializes the value as a quoted string:

```powershell
$parts += "-$($entry.Key)"
$parts += "`"$val`""        # produces: -InstallPostBootCleanup "False"
```

PowerShell accepts `-InstallPostBootCleanup "False"` for a `[bool]` param — the string `"False"` coerces to `$false` correctly. So **this particular case is functionally safe**.

However, `[bool]` params bound via `$PSBoundParameters` only appear in the dictionary when the caller **explicitly** passed `-InstallPostBootCleanup $false`. When the caller passes nothing (default path), `InstallPostBootCleanup` is NOT in `$PSBoundParameters`, so the relaunch args do not include it at all, and the relaunched process re-defaults to `$true`. This is **correct behavior for the default path**.

The actual BLOCKER is: if a CLI caller passes **`-NoPostBootCleanup`** (the opt-out switch), `NoPostBootCleanup` is a `[switch]` and correctly serializes to `-NoPostBootCleanup` in the relaunch args (line 66-67). `InstallPostBootCleanup` stays at its default `$true` and is NOT in `$PSBoundParameters`, so it is not forwarded. The relaunched elevated process receives `-NoPostBootCleanup` **and** `$InstallPostBootCleanup = $true` (default), so the expression `([bool]($InstallPostBootCleanup -and -not $NoPostBootCleanup))` at line 160 evaluates correctly to `$false`. **This is correct.**

Reassessing: there is NO blocker for the GUI path through this function, and no blocker for the headless path through this function when the user passes `-NoPostBootCleanup`. The only remaining concern is described under W1 below.

**Revised severity: WARNING** (see W1 — the `[bool]` param style produces confusing serialization; see also W3 for the missing `.PARAMETER` docs that obscure this).

### WARNING

#### W1 — `[bool]` param style for `$InstallPostBootCleanup` forces "pass explicitly or get default" semantics instead of symmetrical switch pair

**Files:** `tiny11maker.ps1:46`, `tiny11maker-from-config.ps1:25`, `tiny11Coremaker-from-config.ps1:28`

All three wrappers declare:

```powershell
[bool]$InstallPostBootCleanup = $true,
[switch]$NoPostBootCleanup
```

Using `[bool]` means:
- `$PSBoundParameters` only contains `InstallPostBootCleanup` when the caller **explicitly** passes `-InstallPostBootCleanup $false`. Passing `-InstallPostBootCleanup` alone (no value) is a parse error (unlike a switch).
- A headless caller who wants opt-out must pass either `-InstallPostBootCleanup $false` (verbose, fragile quoting on some shells) or `-NoPostBootCleanup` (clean).
- The self-elevation relaunch in `tiny11maker.ps1:59-74` serializes the `[bool]` as a quoted string, not a switch. This is functionally correct but brittle: if someone refactors the `else` branch, the `"False"` string might not be re-parsed. A `[switch]$NoPostBootCleanup` + no `[bool]` pair would be cleaner.

The design is internally consistent across all three wrappers (all three use the same mixed `[bool]` + `[switch]` pattern), but asymmetric with every other toggleable in the param blocks (which are pure `[switch]`).

**Severity: WARNING** — No current production breakage. Risk is in refactoring confusion and the relaunch serialization fragility.

#### W2 — No mutual-exclusivity guard: passing BOTH `-InstallPostBootCleanup $true` and `-NoPostBootCleanup` is silently resolved by the expression, not validated

**Files:** `tiny11maker.ps1:160`, `tiny11maker-from-config.ps1:120`, `tiny11Coremaker-from-config.ps1:117`

All three wrappers resolve the conflict via the expression:

```powershell
([bool]($InstallPostBootCleanup -and -not $NoPostBootCleanup))
```

`-NoPostBootCleanup` wins: if the switch is present, the result is `$false` regardless of `$InstallPostBootCleanup`. This is documented-by-expression only. There is no `[ValidateScript]`, `[Parameter(ParameterSetName)]` separation, or explicit throw for the contradictory case. A caller passing both receives no error and silently gets "skip cleanup" semantics.

This is fragile if a downstream caller ever changes the expression logic without realizing the two-param interaction. It is also undocumented (see INFO-1 / INFO-2).

**Severity: WARNING** — Not a current BLOCKER (behavior is deterministic), but the silent resolution is a footgun.

#### W3 — `tiny11maker.ps1` help block does not document `-InstallPostBootCleanup` or `-NoPostBootCleanup`

**File:** `tiny11maker.ps1:1-35` (`.SYNOPSIS` / `.DESCRIPTION` / `.PARAMETER` block)

The two new params added for v1.0.1 have no `.PARAMETER` stanzas. The existing help block documents `Source`, `Config`, `ImageIndex`, `Edition`, `ScratchDir`, `OutputPath`, `NonInteractive`, and `Internal`. `FastBuild`, `InstallPostBootCleanup`, and `NoPostBootCleanup` are all undocumented.

**Severity: WARNING** — Causes `Get-Help tiny11maker.ps1 -Full` to omit the new flags. A headless user cannot discover the opt-out mechanism from built-in help. `FastBuild` is also missing, but that predates this audit.

#### INFO-2 — `Tiny11.Worker.psm1:Invoke-Tiny11BuildPipeline` param `$InstallPostBootCleanup` is optional with default `$true`; `Tiny11.Core.psm1:Invoke-Tiny11CoreBuildPipeline` follows the same pattern

**Files:** `src/Tiny11.Worker.psm1:45`, `src/Tiny11.Core.psm1:1089`

Both pipeline functions declare:

```powershell
[Parameter()][bool]$InstallPostBootCleanup = $true   # Worker
[bool]$InstallPostBootCleanup = $true                 # Core
```

The Core version omits `[Parameter()]` explicitly; it is still a valid optional parameter. The parameter is not `[Parameter(Mandatory)]`, meaning a caller who accidentally omits it gets silent default-`$true`. This is the intended behavior for legacy callers, and it is consistent with the wrappers. However, the two pipeline functions use slightly different annotation styles for the same parameter — Core drops `[Parameter()]`. Cosmetic inconsistency only.

**Severity: INFO**

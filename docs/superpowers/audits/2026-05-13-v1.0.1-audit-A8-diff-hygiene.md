### WARNING-1: CHANGELOG xUnit test count math is internally inconsistent

**Location:** `CHANGELOG.md`, `[1.0.1]` section, two places:

- "**xUnit:** 4 new BuildHandlers tests ... Total xUnit 80 -> 82." (Added section)
- "xUnit 80 -> 82 (+2 BuildStandardArgs + 2 BuildCoreArgs payload-handling cases)." (Test counts section)

**What is actually in the diff:** `launcher/Tests/BuildHandlersTests.cs` gains exactly **4** new `[Fact]` methods:
- `BuildStandardArgs_OmitsNoPostBootCleanup_WhenEnabled`
- `BuildStandardArgs_AppendsNoPostBootCleanup_WhenDisabled`
- `BuildCoreArgs_OmitsNoPostBootCleanup_WhenEnabled`
- `BuildCoreArgs_AppendsNoPostBootCleanup_WhenDisabled`

Main-branch baseline is 80 xUnit tests (confirmed via `git show main:CHANGELOG.md`). `git show main:launcher/Tests/BuildHandlersTests.cs` has 18 `[Fact]` attributes; HEAD has 22 (+4). Correct total is **84**, not 82.

The statement "80 -> 82 (+2 BuildStandardArgs + 2 BuildCoreArgs)" is self-contradictory: "+2 ... + 2" = +4, yet the delta is written as +2. Likely a copy-paste error from an earlier draft where only 2 of the 4 cases had been committed.

**Severity:** WARNING. No code is wrong; only the documentation count is off. Pre-release fix: update both CHANGELOG occurrences to read "80 -> 84 (+4 BuildHandlers cases)".

### INFO-1: Em-dash -> double-dash substitution in a heredoc comment in `src/Tiny11.Core.psm1`

**Location:** `src/Tiny11.Core.psm1`, hunk around line 680 (context: `New-Tiny11CorePostBootCleanupScript` refactor):

```diff
-:: The .ps1 and .xml files in the same directory STAY — the scheduled task references the .ps1
+:: The .ps1 and .xml files in the same directory STAY -- the scheduled task references the .ps1
```

The em-dash (`—`, U+2014) was replaced with a double-hyphen (`--`). This is a CMD batch file comment inside a heredoc. The change is intentional per the ASCII-only-user-scripts policy (CLAUDE.md `feedback_ascii_only_user_scripts.md` -- multi-line scripts must be ASCII-only because em-dashes survive until Windows-1252 mangling breaks them at parse time). Functionally correct; the original line had a non-ASCII character inside a batch script that runs under `cmd.exe`. This substitution was the right fix.

**Severity:** INFO. Confirms the change was intentional and policy-compliant -- no action needed.

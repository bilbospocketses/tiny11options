## Helper function audit

### `Set-RegistryValue` (PostBoot.psm1 lines 52-69)

- Creates missing key with `New-Item -Path $KeyPath -Force` (not `-LiteralPath`) — paths with `[]` characters would fail. Unlikely in real registry paths but technically fragile. WARNING.
- Idempotency check: `$current -eq $Value`. For `[string[]]` (REG_MULTI_SZ), PS `-eq` between arrays returns an array of matching elements, not a scalar bool. This does NOT throw under `$ErrorActionPreference = 'Continue'` (no StrictMode in generated script), but the condition is never true, so it always re-writes. Functionally tolerable but semantically wrong. WARNING.
- Uses `Set-ItemProperty -Type $Type` — correct for all standard types.

### `Set-RegistryValueForAllUsers` (PostBoot.psm1 lines 71-79)

- Enumerates `HKU:\` for SIDs matching `^S-1-5-21-` (domain/local user SIDs) — correct.
- Appends `'.DEFAULT'` unconditionally.
- **`HKU:\.DEFAULT` is NOT the default new-user profile hive.** It is the hive for Local Service / Network Service / unelevated SYSTEM operations. New user profiles inherit from `C:\Users\Default\NTUSER.DAT`. Writing to `.DEFAULT` does NOT cause CDM (or any NTUSER-scope) setting to be inherited by newly created user accounts. **This is a BLOCKER** for any catalog item whose goal is to apply a per-user setting to all future users. The correct approach is to also load and write `C:\Users\Default\NTUSER.DAT` via `reg load`.
- The same issue applies to `Remove-RegistryKeyForAllUsers` (lines 95-103).

### `Remove-AppxByPackagePrefix` (PostBoot.psm1 lines 141-170)

- Uses `Get-AppxProvisionedPackage`, `Remove-AppxProvisionedPackage`, `Get-AppxPackage -AllUsers`, `Remove-AppxPackage -AllUsers`.
- These cmdlets live in the `Appx` (or `Appx` alias module) on Windows. In PS 5.1, the `Appx` module is **not auto-loaded in a SYSTEM-context scheduled task on a debloated image** — it requires either `Import-Module Appx` or `Import-Module Appx -UseWindowsPowerShell` (PS 7). There is **no `Import-Module Appx` anywhere** in the helpers block or the generator. **This is a BLOCKER** on any debloated image where module auto-discovery is impaired, or on any server SKU without the Desktop Experience feature.
- The function uses `Get-AppxProvisionedPackage -Online` which requires elevation (SYSTEM has it) and the `Appx` module.

## Online-command emitter audit

### Registry (`Tiny11.Actions.Registry.psm1`)

- **`DEFAULT` hive maps to `HKU:\.DEFAULT` via a direct (non-fan-out) `Set-RegistryValue` call** (line 35: `'DEFAULT' { 'HKU:\.DEFAULT' }`). This is the same `.DEFAULT` BLOCKER described above. If a catalog item uses `hive: DEFAULT` and the intent is "default new-user profile", this is wrong. The fan-out path (`NTUSER`) correctly calls `Set-RegistryValueForAllUsers`.
- `COMPONENTS` hive: correctly throws with an explanatory message — good.
- REG_BINARY: emits `[byte[]]` parsed via `(-split ... | ForEach-Object { [Convert]::ToByte($_, 16) })`. The `,()` unary operator correctly wraps the byte array to prevent pipeline unwrapping (line 55).
- REG_MULTI_SZ: similar `,()` wrapping (line 56). Correct.
- Output: single `[pscustomobject]` wrapped in `,()` for each op. Returns one command object per action. Correct.
- Missing required field guard: none. If `$Action.name` or `$Action.key` is missing, the switch falls through to string interpolation which silently inserts empty string. WARNING.

### Filesystem (`Tiny11.Actions.Filesystem.psm1`)

- `Get-Tiny11FilesystemOnlineCommand` lines 39-53.
- Path emitted as `'$env:SystemDrive\' + $Action.path` (line 51). Because this is a bare PowerShell string concatenation at generator time, the generated script will contain the literal text `$env:SystemDrive\<path>` inside single quotes. Wait — checking more carefully: `'$env:SystemDrive\' + $Action.path` — the first part is a single-quoted string literal `$env:SystemDrive\`, concatenated with `$Action.path`. The `Args` hashtable value is therefore a plain string like `$env:SystemDrive\Windows\temp\somefile`. When `Format-PSNamedParams` processes it, it single-quotes the whole thing: `'$env:SystemDrive\Windows\temp\somefile'`. Single quotes suppress variable expansion — **`$env:SystemDrive` will NOT expand at runtime.** **This is a BLOCKER.** The path argument should use a double-quoted string or be assembled differently so that `$env:SystemDrive` expands at runtime in the generated PS script. Correct emission would be: `-Path "$env:SystemDrive\Windows\temp\somefile"`.
- `Recurse` field: `[bool]$Action.recurse` — correctly cast from potentially-JSON-null or missing field (missing field gives `$false` on cast). INFO.

### ScheduledTask (`Tiny11.Actions.ScheduledTask.psm1`)

- `Get-Tiny11ScheduledTaskOnlineCommand` lines 18-30.
- Same `'$env:SystemRoot\System32\Tasks\' + $relPath` pattern — same BLOCKER: `$env:SystemRoot` will not expand at runtime when single-quoted by `Format-PSNamedParams`.
- Only supports `op: remove`. Correct (tasks are removed, not created, via catalog actions).
- `Recurse` field from `$Action.recurse` — same cast consideration as filesystem.

### ProvisionedAppx (`Tiny11.Actions.ProvisionedAppx.psm1`)

- `Get-Tiny11ProvisionedAppxOnlineCommand` lines 42-51.
- Emits `Remove-AppxByPackagePrefix -Prefix '<packagePrefix>'`. No env-var path issues (prefix is a plain string). Correct.
- No validation of `$Action.packagePrefix` for null/empty — would emit `Remove-AppxByPackagePrefix -Prefix ''` which would match all packages. WARNING.

## Script structural integrity (generated script)

- `$ErrorActionPreference = 'Continue'` set in header block (line 22). Correct — individual action failures continue the run.
- **No `Set-StrictMode` in generated script.** The generator module itself has `Set-StrictMode -Version Latest` but this does NOT propagate into the generated script string. The generated script runs without strict mode. This means uninitialized variables and method-call-on-null are silently forgiven at runtime. INFO (could mask bugs).
- Individual action calls (emitted at lines 222-224) are **bare calls with no try/catch wrapper.** The helper functions themselves have internal try/catch, so most failures are logged. But if a helper function throws despite its own guard, `$ErrorActionPreference = 'Continue'` will write to error stream and continue. Functionally acceptable.
- End-of-file: `Write-CleanupLog '==== tiny11-cleanup done ===='` (footer block line 174). No explicit `exit` or `exit 0`. The task scheduler captures exit code from `powershell.exe` which will be `0` unless the script throws an unhandled terminating error. Acceptable.
- **PS 5.1 syntax compatibility:** All constructs used (here-strings, ForEach-Object, Where-Object, `[ordered]@{}`, `Set-StrictMode -Version Latest`) are compatible with PS 5.1. No PS 6+ syntax observed. INFO-positive.

## Findings

### BLOCKER-1: `HKU:\.DEFAULT` does not propagate to new user profiles

**File:** `src/Tiny11.PostBoot.psm1` lines 75-78; `src/Tiny11.Actions.Registry.psm1` line 35

`Set-RegistryValueForAllUsers` and `Remove-RegistryKeyForAllUsers` both append `'.DEFAULT'` to the SID list. `HKU:\.DEFAULT` maps to `HKEY_USERS\.DEFAULT` — the hive used by Local Service and Network Service accounts, and as a pre-login placeholder. It is **not** the template for new user profile creation. New user profiles are copied from `C:\Users\Default\NTUSER.DAT`. Any NTUSER-scope catalog item (CDM disablement, telemetry settings, etc.) that targets `.DEFAULT` **will not be inherited by users created after first boot.** Fix: additionally load `C:\Users\Default\NTUSER.DAT` via `reg load HKLU:\Default C:\Users\Default\NTUSER.DAT`, write the value, then `reg unload HKLU:\Default`.

### BLOCKER-2: `$env:SystemDrive` and `$env:SystemRoot` are single-quoted in generated script

**File:** `src/Tiny11.Actions.Filesystem.psm1` line 51; `src/Tiny11.Actions.ScheduledTask.psm1` line 26

Both emitters build the `Path` argument value as a plain string that begins with the literal text `$env:SystemDrive\` or `$env:SystemRoot\`. When `Format-PSNamedParams` processes this value, it wraps it in single quotes: `'$env:SystemDrive\Windows\temp\file'`. In PowerShell, single-quoted strings do not expand variables. At runtime, the cleanup script will attempt to use the literal string `$env:SystemDrive\Windows\temp\file` as a path, which does not exist. **Every filesystem and scheduled-task action will silently no-op** (the helpers check `Test-Path -LiteralPath $Path` and return "absent (no-op)" on failure). Fix: emit paths using a double-quoted token or emit the path with explicit expansion: `Args = [ordered]@{ Path = "`$env:SystemDrive\$($Action.path)"; ... }` — but because `Format-PSNamedParams` always single-quotes strings, the correct fix is to emit the path as a here-string or change `Format-PSNamedParams` to detect and double-quote strings containing `$env:` tokens.

### BLOCKER-3: `Remove-AppxByPackagePrefix` has no `Import-Module Appx`

**File:** `src/Tiny11.PostBoot.psm1` lines 141-170 (helpersBlock)

`Get-AppxProvisionedPackage`, `Remove-AppxProvisionedPackage`, `Get-AppxPackage`, and `Remove-AppxPackage` are all members of the `Appx` module. In Windows PowerShell 5.1, the `Appx` module is available on client SKUs but is **not guaranteed to auto-load** in a SYSTEM-context scheduled task on a debloated or server image. The generated cleanup script has no `Import-Module Appx` call. On a standard Windows 11 consumer image this will likely work due to module auto-discovery, but on any image where the Appx module path is not in `$env:PSModulePath` for SYSTEM, all Appx removal calls will fail with "The term 'Get-AppxProvisionedPackage' is not recognized." Fix: add `Import-Module Appx -ErrorAction Stop` at the top of `Remove-AppxByPackagePrefix`, or add it to the helpers block header.

### WARNING-1: No `RestartOnFailure` in scheduled task XML

**File:** `src/Tiny11.PostBoot.psm1` lines 264-274 (`<Settings>` block)

The `<Settings>` block has no `<RestartOnFailure>` element. Spec called for retry 3 times at 1-minute intervals. Without this, if the task fails (e.g., locked files at boot), it will not retry until the next trigger fires. Add inside `<Settings>`: `<RestartOnFailure><Interval>PT1M</Interval><Count>3</Count></RestartOnFailure>`.

### WARNING-2: Unknown action type terminates generator without error context

**File:** `src/Tiny11.Actions.psm1` line 28; `src/Tiny11.PostBoot.psm1` lines 219-224

`Get-Tiny11ActionOnlineCommand` throws on unknown `$Action.type`. `New-Tiny11PostBootCleanupScript` has no try/catch around this call. A single malformed catalog action will abort the entire generator and return nothing. The caller (`Install-Tiny11PostBootCleanup`) also has no guard; it will write a null/empty string to `tiny11-cleanup.ps1`. The error message does not include the parent item ID or action index. Fix: wrap in try/catch within the generator loop; log the offending item ID and action; skip or rethrow with context.

### WARNING-3: REG_MULTI_SZ idempotency check is broken

**File:** `src/Tiny11.PostBoot.psm1` line 59

`if ($null -ne $current -and $current -eq $Value)` — when `$current` is a `[string[]]` array and `$Value` is also a `[string[]]` array, PowerShell's `-eq` operator performs an element-wise filter and returns an array (possibly empty). This is truthy/falsy based on array length, not semantic equality. The idempotency check will behave incorrectly for REG_MULTI_SZ values: it may always evaluate as falsy (if the filter returns empty array), causing unnecessary re-writes on every run, or could produce unexpected behavior. Fix: use `[System.Linq.Enumerable]::SequenceEqual($current, $Value)` or `(Compare-Object $current $Value).Count -eq 0` for array comparison.

### WARNING-4: Missing `$Action.packagePrefix` null/empty guard

**File:** `src/Tiny11.Actions.ProvisionedAppx.psm1` lines 42-51

`Get-Tiny11ProvisionedAppxOnlineCommand` uses `$Action.packagePrefix` without checking for null or empty. If a catalog item's provisioned-appx action has an empty or missing `packagePrefix`, the generated call `Remove-AppxByPackagePrefix -Prefix ''` will match ALL provisioned packages whose DisplayName starts with empty string — effectively all packages — and remove everything. The offline action `Invoke-ProvisionedAppxAction` has the same issue (line 38: `$_ -like "*$($Action.packagePrefix)*"` with empty prefix matches all). Fix: validate `$Action.packagePrefix` is non-empty in both emitter and dispatcher.

### WARNING-5: `New-Item -Path` (not `-LiteralPath`) in `Set-RegistryValue`

**File:** `src/Tiny11.PostBoot.psm1` line 55

`New-Item -Path $KeyPath -Force` — uses positional `-Path` (not `-LiteralPath`). Registry key paths containing `[` or `]` characters would be misinterpreted as wildcard patterns. Registry key names can legally contain brackets. Use `-LiteralPath` for consistency with the rest of the function.

### WARNING-6: `$Catalog` null guard absent in generator

**File:** `src/Tiny11.PostBoot.psm1` line 213

`foreach ($item in $Catalog.Items)` — if `$Catalog` is `$null`, this throws `NullReferenceException`. The parameter is typed as untyped `$Catalog` (not `[pscustomobject]` or a typed class), so `[Parameter(Mandatory)]` ensures it's provided but not that it has an `.Items` property. Low risk in practice (callers use `Get-Tiny11Catalog` which always returns a proper object), but worth noting.

### INFO-1: Generated script has no `Set-StrictMode`

**File:** `src/Tiny11.PostBoot.psm1` lines 12-49 (headerBlock)

The header block does not include `Set-StrictMode -Version Latest`. The generator module runs strict, but the generated script does not. Uninitialized variables in future helper additions would silently evaluate as `$null`. Low risk for current helpers which are explicit, but a quality gap.

### INFO-2: Dangling item comment for zero-action items

**File:** `src/Tiny11.PostBoot.psm1` lines 217-226

If a catalog item has `actions = []` (valid per schema), the generator emits `# --- Item: DisplayName (id) ---` followed by a blank line but no action calls. The script remains syntactically valid and functionally harmless. Cosmetically noisy in the generated file.

### INFO-3: Task URI is `\tiny11options\Post-Boot Cleanup` not under `\Microsoft\`

**File:** `src/Tiny11.PostBoot.psm1` line 243

`<URI>\tiny11options\Post-Boot Cleanup</URI>`. The audit spec expected the path `Windows\System32\Tasks\Microsoft\tiny11options Post-Boot Cleanup` (under `Microsoft\`). Actual URI omits `\Microsoft\`. This is a naming convention choice, not a functional defect. The `schtasks /tn` in SetupComplete.cmd (line 297) is consistent with the URI. No broken behavior.

### INFO-4: `Format-PSNamedParams` has no `[hashtable]` type branch

**File:** `src/Tiny11.PostBoot.psm1` lines 186-195

No current online-command emitter passes a hashtable value. The function would silently coerce `[hashtable]` to the string `System.Collections.Hashtable`. If a future emitter adds a hashtable-typed parameter, the generated script would be silently broken. Low risk today; worth guarding with a `throw` or `@{}` inline emitter.

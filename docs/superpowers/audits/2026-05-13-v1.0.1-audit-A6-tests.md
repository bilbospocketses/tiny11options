## Hives flaky-test cause analysis

**Hypothesis (CONFIDENT):**

`Tiny11.Hives.Tests.ps1` (line 1-2) loads via `Import-Tiny11Module`, which calls
`Import-Module ... -Force -DisableNameChecking -Global`.  The `-Global` flag promotes
the module into the *session-level* (global) scope rather than the Pester run-scope.

When v1.0.1 introduced `BeforeAll` blocks in many new test files with standalone
`Import-Module ... -Force` (no `-Global`), those `-Force` re-imports reload
`Tiny11.Hives`'s transitive dependencies at module-scope, demoting the previously
`-Global` Hives module back to local scope or replacing it in the session table.
Specifically:

- `Tiny11.PostBoot.Generator.Tests.ps1:BeforeAll` imports `Tiny11.Actions.psm1 -Force`
  (line 6).  `Tiny11.Actions.psm1` itself likely auto-imports `Tiny11.Hives.psm1` as a
  dependency. The `-Force` re-import re-runs the module's body in a fresh scope,
  replacing the previously `-Global`-scoped session entry. When subsequent Hives tests
  run, `Mock -CommandName 'Invoke-RegCommand' -ModuleName 'Tiny11.Hives'` targets the
  new in-scope module instance, not the global one. The module address differs, so
  Pester's mock registry hits the wrong instance and `Should -Invoke` assertions
  either see zero calls (test fails) or the mock itself never intercepts.

- The symptom (passes alone, fails in suite) is the canonical signature of a stale mock
  target: when Hives loads in isolation, there is only one module instance. When
  preceded by a `-Force` import of Actions (which re-loads Hives as a dependency), the
  instance pointer changes.

**Which 7 tests fail in suite (UNCERTAIN - no .fail.log; inferred from structure):**

Likely candidates from `Tiny11.Hives.Tests.ps1` Describe blocks that rely on
`Mock -ModuleName 'Tiny11.Hives'`:

1. `Mount-/Dismount-Tiny11Hive > calls reg load with HKLM\\zSOFTWARE and the resolved path` (line 25)
2. `Mount-/Dismount-Tiny11Hive > calls reg unload with HKLM\\zNTUSER` (line 31)

The `Resolve-Tiny11HivePath` tests (lines 4-21) use no mocks and should be robust.
The 7 failures therefore likely span multiple test files that also use
`Import-Tiny11Module` (via TestHelpers with `-Global`) and then get clobbered:
`Tiny11.Actions.Tests.ps1`, `Tiny11.Worker.Tests.ps1` also use `Import-Tiny11Module -Global`
and set mocks with `-ModuleName`. Any file that subsequently re-imports Actions or Hives
with plain `-Force` can invalidate up to 5 mock-dependent tests per file.

UNCERTAIN: exact 7 names require running `Invoke-Pester` with detailed output.

### BLOCKER-1: No test asserts that PostBoot.psm1 (or src/Tiny11.PostBoot.psm1) is embedded as a StaticResource

**File:** `launcher/Tests/EmbeddedResourcesTests.cs:45-55`

`EmbeddedResourcesTests.RealResource_IsEmbedded` is parameterized with:
- `tiny11maker.ps1`
- `autounattend.template.xml`
- `src/Tiny11.Iso.psm1`
- `ui/index.html`

There is no `[InlineData("src/Tiny11.PostBoot.psm1")]` entry (nor any Workers/Actions
chain entries). Regression b35d73a was exactly this: the `.csproj` EmbeddedResource glob
missed `PostBoot.psm1`, HeadlessRunner extracted resources into a temp dir, and the
PostBoot module was absent causing a runtime failure invisible to the test suite.

**Impact:** A future `.csproj` edit that accidentally excludes `PostBoot.psm1` from
`<EmbeddedResource>` globs would not be caught by any test.

**Fix direction:** Add `[InlineData("src/Tiny11.PostBoot.psm1")]` and
`[InlineData("src/Tiny11.Actions.psm1")]` (and Worker) to `RealResource_IsEmbedded`.

### BLOCKER-2: No integration test for the -Force cascade demoting Hives global scope (regression bdb38d7)

**Files:** All PostBoot test files + `Tiny11.Worker.Tests.ps1`

There is no test that:
1. Loads the full production import chain (Worker -> Actions -> PostBoot) using the
   mixed `-Global` + standalone `-Force` pattern that production uses, and
2. Then calls a Hives-dependent function and asserts it succeeds.

`Tiny11.Worker.Tests.ps1:40-52` tests that `Invoke-Tiny11BuildPipeline` has the
`InstallPostBootCleanup` parameter and that source contains the right calls -- but it
does so by reading source text (`Get-Content ... -Raw`), not by actually running the
pipeline with PostBoot enabled. A text-pattern test cannot catch scope-demotion
regressions.

**Impact:** If a future `-Force` import in any test file or the production load sequence
re-demotes Hives, the full pipeline will silently fail at hive-mount time. No test will
catch this before it ships.

**Fix direction:** Add an integration `It` in `Tiny11.Worker.Tests.ps1` (or a new
`Tiny11.Worker.Integration.Tests.ps1`) that:
- Imports the three modules in production order (Worker -> Actions -> PostBoot) using
  the same flag mix as production
- Calls `Resolve-Tiny11HivePath` (or `Mount-Tiny11Hive` with a mock) immediately
  after PostBoot import
- Asserts no error (i.e., the function is still callable)

### WARNING-1: Global-scope leakage from TestHelpers vs. standalone -Force creates silent mock-target mismatch

**Files:**
- `tests/Tiny11.TestHelpers.psm1:7` -- `Import-Tiny11Module` always uses `-Force -Global`
- `tests/Tiny11.Actions.Tests.ps1:1-2` -- uses `Import-Tiny11Module` (-> `-Global`)
- `tests/Tiny11.Worker.Tests.ps1:1-2` -- uses `Import-Tiny11Module` (-> `-Global`)
- `tests/Tiny11.Hives.Tests.ps1:1-2` -- uses `Import-Tiny11Module` (-> `-Global`)
- All PostBoot + Online test files -- use standalone `Import-Module ... -Force` (no `-Global`)

Two distinct import protocols coexist. Files using the TestHelpers path get their
modules at global scope; files using the standalone path get them at local (Pester
run) scope. When Pester runs the full suite in order, a standalone `-Force` import
re-registers the module at local scope, silently displacing the global instance that
mocks were registered against.

This is the proximate mechanism for the 7 Hives flakies.

**v1.0.1 made this worse:** All 12 new test files use standalone `-Force` without
`-Global`. Each one that imports a module also depended on by a TestHelpers-style
test creates a fresh instance collision opportunity. The Online tests
(`Tiny11.Actions.Registry.Online.Tests.ps1`, etc.) import `Tiny11.Actions.Registry.psm1`
independently; when Actions.Tests.ps1 subsequently runs with its `-Global` import,
the mock registry has a dangling reference.

**Pester version note:** This is a Pester 5 behavior. In Pester 5, `-Global` import
is explicitly discouraged in favor of `BeforeAll { Import-Module ... }` scoped imports.
The TestHelpers pattern using `-Global` is a Pester 4 holdover.

### WARNING-2: Golden fixture comparison is byte-equal (brittle to line-ending changes)

**File:** `tests/Tiny11.PostBoot.Helpers.Golden.Tests.ps1:42-47`

```powershell
$golden = [System.IO.File]::ReadAllText($goldenPath)
$script:helpers | Should -Be $golden
```

`[System.IO.File]::ReadAllText` uses the OS default encoding and preserves raw bytes.
`$script:helpers` is a PowerShell string (always LF on any OS). If `git` or an editor
rewrites the golden fixture with CRLF line endings (Windows default), the comparison
will fail spuriously. Conversely if the production code ever changes to CRLF output,
the golden passes on Windows but fails on Linux CI.

There is also no `.gitattributes` `eol` override on `tests/golden/*.txt` observed in
the files read (UNCERTAIN -- `.gitattributes` not read in this audit). If the file
lacks an explicit `text eol=lf` rule, `git checkout` on Windows may silently CRLF it.

**Fix direction:** Normalize the comparison: `$golden.Replace("`r`n", "`n")` and
`$script:helpers.Replace("`r`n", "`n")` before `Should -Be`. Or add `eol=lf` to
`.gitattributes` for `tests/golden/*.txt`.

### WARNING-3: No BeforeAll module import in Tiny11.Hives.Tests.ps1 (file-level import)

**File:** `tests/Tiny11.Hives.Tests.ps1:1-2`

```powershell
Import-Module "$PSScriptRoot/Tiny11.TestHelpers.psm1" -Force
Import-Tiny11Module -Name 'Tiny11.Hives'
```

These are file-level statements outside any `Describe` / `BeforeAll` block. In Pester 5,
file-level code executes at discovery time (the script is dot-sourced during collection),
which means the import fires *before* Pester has set up the run context. This is not
merely style drift -- it means the mock context established by the subsequent
`BeforeEach { Mock ... }` (line 24) is operating against a module that was imported
outside any Pester scope boundary. Any Pester 5 version that tightens scope enforcement
(e.g., strict-mode container isolation) could cause these to silently not mock.

The same pattern appears in:
- `tests/Tiny11.Actions.Tests.ps1:1-2`
- `tests/Tiny11.Worker.Tests.ps1:1-2`

**Fix direction:** Wrap the `Import-Module` / `Import-Tiny11Module` calls in `BeforeAll`
blocks at the `Describe` level. Migrate from TestHelpers `-Global` pattern to
Pester 5 scoped imports.

### WARNING-4: Pester version not pinned in config; only floor-checked

**File:** `tests/Run-Tests.ps1:1`

```powershell
#Requires -Module @{ ModuleName='Pester'; ModuleVersion='5.0.0' }
```

This enforces Pester >= 5.0.0 but does not pin the exact minor/patch. Pester 5.x has
had several breaking changes in mock scoping (e.g., 5.3 changed BeforeAll variable
visibility). Running on Pester 5.6 vs 5.0 can change whether file-level
`Import-Tiny11Module` calls are visible inside `Describe` blocks. The TestHelpers
`-Global` workaround was almost certainly written to compensate for a specific 5.x
version's scope behavior.

**Fix direction:** Pin to a specific Pester release in `#Requires` and/or in a
`requirements.psd1` / `pester.psd1` manifest. At minimum document the tested version.

### INFO-1: No AfterAll Remove-Module in any test file (module leak by design, but undocumented)

**Files:** All Pester test files

No test file calls `Remove-Module` in `AfterAll`. This is intentional in Pester 5
(module cleanup is not required when using scoped imports), but combined with the
TestHelpers `-Global` import pattern, modules *persist* across all subsequent test
files in the same `Invoke-Pester` run. This is the surface that makes leakage possible
and is documented here for completeness. It is not a bug in itself given the current
architecture, but becomes one if any new test assumes a clean slate.

### INFO-2: Tiny11.PostBoot.* tests load PostBoot.psm1 without Worker or Actions chain

**Files:** All 7 `Tiny11.PostBoot.*.Tests.ps1` files

Every PostBoot test loads `Tiny11.PostBoot.psm1` directly:
```powershell
Import-Module (Join-Path $PSScriptRoot '..' 'src' 'Tiny11.PostBoot.psm1') -Force -DisableNameChecking
```

In production, PostBoot is loaded only after Worker -> Actions. If PostBoot has any
`using module` or dot-source dependency on Actions or Worker internals, isolated load
would succeed but production-integrated load could fail. Based on the tests read,
PostBoot appears self-contained (pure output formatting + script generation), so
this is low-risk INFO today. The exception is `Tiny11.PostBoot.Generator.Tests.ps1`
(line 6-8) which does load Actions + PostBoot + Selections, approximating the
production chain, but still without Worker.

### INFO-3: xUnit BuildHandlersTests - no static state leakage; contract assertions are correctly structural

**File:** `launcher/Tests/BuildHandlersTests.cs`

xUnit creates a new `BuildHandlersTests` instance per `[Fact]`. `BuildHandlers` is
instantiated fresh in each test body (`var bh = new BuildHandlers(bridge, ...)`), so
there is no cross-test static state leakage. The `BuildCoreArgs`/`BuildStandardArgs`
tests assert on string contents (`Assert.Contains("-NoPostBootCleanup", result)`) rather
than the full command-line string -- this is correct; substring matching is tolerant of
ordering changes and additional flags. `DismountSourceIsoIfApplicable_EarlyReturnsCompletedTask`
uses a 250ms ceiling which is generous but could theoretically flake on a heavily loaded
CI box; however this is low-risk given the test exercises a near-instant early-return path.
No issues found in the xUnit suite.

# v1.0.9 UI Redesign — Step 1 + Step 3 — Design Spec

**Date:** 2026-05-16
**Target release:** v1.0.9
**Branch:** `feat/v1.0.9-ui-redesign-steps-1-3`
**Status:** design approved; pending implementation plan

## Problem

After v1.0.8 shipped (audit-cleanup release, 32 fixes), user feedback called for layout refinement on the wizard's Step 1 (source/build-options) and Step 3 (ready-to-build/progress). Step 2 (customize) was explicitly judged "great as designed" and is **out of scope**.

The specific Step 1 pain: toggling the **Build tiny11 Core** checkbox reflows the bottom of the screen (a warning panel and a `.NET 3.5` checkbox appear/disappear), causing visual instability when the user is mid-decision in the middle of the form.

The specific Step 3 pain: the Output ISO field is an editable input awkwardly jammed inside an otherwise read-only `<dl>` summary table. Every other row on the screen is read-only text — Output ISO is the lone editable field — but it visually blends in with the read-only rows. The user wants a fully read-only summary on Step 3 with explicit "edit in Step 1" navigation links, both as clickable breadcrumb steps in the header and as per-row links inside the summary.

## Decisions locked during brainstorming

| Decision | Outcome |
|---|---|
| Refinement scope | Structural reorg within the wizard. 3-step wizard contract (Source / Customize / Build) and Step 1/3 boundaries preserved. Cosmetic-only rejected (too small a delta); workflow re-thinks rejected (too much surface area, risks regressions in wizard plumbing). |
| Step 1 layout direction | **Two-column** (CSS Grid). Required inputs in the left column; build options in the right column. Core mode toggle relocated to the bottom of the right column so its conditional drawer expansion is structurally trapped in that column — required inputs literally cannot reflow. |
| Step 3 layout direction | **Segmented confirmation** — four bordered cards stacked vertically (Paths / Build mode / Customizations / Ready to build). Each summary card has its own header-level "Edit in Step X" link. CTA card last. |
| Output ISO field location | **Moves from Step 3 to Step 1** (left column, below Scratch directory). Step 3 becomes pure confirmation — no editable fields. |
| Required fields on Step 1 | All four left-column fields (Source ISO, Edition, Scratch directory, Output ISO) are required. Forward nav (Next button + forward breadcrumb links to Steps 2 + 3) is gated until all four are non-empty. |
| Scratch directory default | Auto-fill on first render via a new `window.__autoScratchPath` injection from C# (resolves `%TEMP%` + random 8-char suffix at app launch). User can overwrite. Field is never empty visually — the "all required" rule is satisfied without breaking today's no-typing convenience. |
| Output ISO default | Continues to auto-fill via the existing `prefillOutputIfEmpty()` chain (now triggered at boot init because scratchDir is boot-populated). User overrides preserved via new `state.outputPathIsAuto` boolean flag. |
| Breadcrumb interaction | Steps become `<button>` elements with click handlers. Backward nav always allowed. Forward nav uses the same gating predicate as the Next button. Current step uses `aria-current="step"`. Gated forward links use `aria-disabled="true"` (not `disabled`) so screen readers still announce them with the describedby reason. |
| Step 3 state transformations | Cards 1–3 stay visible (frozen at click-time values) during the building / complete / error states. Only Card 4 transforms in place between idle → progress → complete → error. Partial fix for "4 stitched screens" feel without restructuring the underlying state machine. |
| Hint copy treatment | Stays as inline paragraphs under each Step 1 option. Right column gets to grow — that's the whole point of two-column. No tooltip-ification or density reduction. |
| Visual indicators for required | Small red `*` after each label (`aria-hidden="true"`) + `aria-required="true"` on the input. No live red-border-of-shame on initial empty state. |
| Animations | No drawer animation (render-on-state model re-renders on toggle). Cleanup banner preserves today's sticky behavior (no auto-fade) to keep this cycle visual-only. |
| Out of scope explicitly | Step 2 layout; density reduction / tooltip-ification of hint copy; workflow re-thinks; JS test harness addition; Trusted Signing (v1.0.10); any new features. |

  [____________] [Browse...]            [x] Fast build (skip recovery compression)

  [Windows 11 Pro ▾]                        <hint paragraph>

  […\tiny11.iso] [Browse...]            [ ] Build tiny11 Core (smaller, non-serviceable)

- Order in left column: ISO → status line → Edition → Scratch → Output ISO.
- Order in right column: Unmount → Fast build → Post-boot → Log build (+ indented Append) → Core mode (last; drawer expands below within same column).
- Both columns top-aligned (`align-items: start`); right column free to grow without affecting left.

### Step 3 state transformations (Card 4 only; Cards 1–3 stay frozen)

| State | Card 4 contents |
|---|---|
| **Idle** | Hint line + `Build ISO` primary button + output-path footnote |
| **Building** | `<progress>` bar + Phase + Step + `Cancel build` + `Cancel build & clean up` + build-details accordion (today's `renderProgress` body, hosted inside the card) |
| **Complete** | Green ✓ + `Build complete` + output path + `Open output folder` + `Close` + (optional) completion-cleanup block |
| **Error** | Red ✗ + `Build failed` or `Build cancelled` header + message + log path + cleanup block + `Close` |

Cards 1–3 are read-only history of what was/is being built; their values freeze at the moment the user clicks Build ISO.

### Breadcrumb navigation

`<nav class="breadcrumb" aria-label="Wizard progress">` with three `<button>` children. Three visual states per button:

| State | Style | Behavior |
|---|---|---|
| Current step (`aria-current="step"`) | accent background, white text, no hover affordance | non-clickable |
| Reachable step | accent-colored text, underline-on-hover | click → `goToStep(target)` |
| Gated forward step | muted text, no underline, `cursor: not-allowed`, `aria-disabled="true"`, `aria-describedby="forward-nav-gate-reason"` | click → no-op |

**Gating predicate** (factored into `canMoveForward()`, called identically by Next button and breadcrumb forward links):

```
canMoveForward = sourceFilled && editionFilled && scratchFilled && outputFilled
```

where `*Filled` is `(value || '').trim().length > 0`.

**Reachability matrix:**

| On step | Reachable backward | Reachable forward |
|---|---|---|
| 1 | (none) | 2 + 3 (gated by `canMoveForward`) |
| 2 | 1 (always) | 3 (always — by construction Step 1 was complete to reach Step 2) |
| 3 | 1 + 2 (always) | (none) |

If a user navigates back from Step 3 → Step 1, clears Output ISO, and then clicks "3. Build" — the click is gated and shows the disabled style + tooltip until they re-fill the field.

**Gated tooltip / describedby text:** `Fill in all required Source & paths fields first.` Visually-hidden `<span id="forward-nav-gate-reason">` holds it; Next button + both forward breadcrumb links share the single string.

### Output ISO auto-fill behavior

New state flag: `state.outputPathIsAuto` (default `true`).

- `prefillOutputIfEmpty()` runs at boot init (scratchDir is now boot-populated via `__autoScratchPath`) and on any scratchDir change. Writes to outputPath **only if** `state.outputPathIsAuto`. Computed value: `<scratchDir>\tiny11.iso` (or `tiny11core.iso` in Core mode).
- `syncOutputFilenameToMode()` runs on Core mode toggle. Swaps `tiny11.iso` ↔ `tiny11core.iso` **only if** `state.outputPathIsAuto`.
- User typing in Output ISO field or choosing via Browse flips `outputPathIsAuto = false`.
- Profile load resets `outputPathIsAuto = false` (profile values are explicit user choices, never to be overwritten).

Replaces today's brittle filename-string-matching heuristic (`if (outputPath ends with 'tiny11.iso' || 'tiny11core.iso')`) with one explicit boolean.

### Cleanup status banner (Step 3)

When `state.cleanupStatus` is non-null, full-width banner above Card 1. Three states (same visual semantics as today's `renderInlineCleanupStatus()`):

- `kind: 'progress'` — spinner + message
- `kind: 'success'` — green ✓ + message (sticky until user navigates away — today's behavior preserved)
- `kind: 'error'` — red ✗ + message + `Retry cleanup` button

`outputWarning` block from today's `renderBuildStep` is **deleted** — Output ISO empty is no longer reachable (Step 1's required gate catches it).

### Accessibility

- Required fields: visual asterisk (`aria-hidden`) + `aria-required="true"` on input. Screen reader: "Source, edit, required."
- Gated forward controls: `aria-disabled="true"` + `aria-describedby` pointing to the visually-hidden gate-reason text.
- Breadcrumb: `<nav aria-label="Wizard progress">` landmark; each step button gets `aria-current="step"` when active.
- Step 3 edit links: `<button aria-label="Edit Paths in Step 1">` (etc.). Click handler:
  1. `state.step = <target>`
  2. `renderStep()`
  3. After render commits, focus the target field (per-link selector via `data-edit-target` attribute)
- Focus-on-edit targets:
  - Paths → Source ISO input
  - Build mode → Unmount source checkbox (first option in right column)
  - Customizations → Step 2 search input
- Tab order: theme/update buttons → breadcrumb steps left-to-right → main content → footer (Back / Next).

### Theming

All new CSS uses variables defined in both `[data-theme="light"]` and `[data-theme="dark"]` blocks at the top of `style.css`. No hardcoded hex in new code (extends v1.0.8 audit `ui B7` rule). New tokens:

- `--card-border` / `--card-bg` / `--card-radius`
- `--section-heading-color` / `--section-heading-tracking`
- `--req-asterisk-color` (red, WCAG AA contrast in both themes)
- `--link-disabled-color` (muted in both themes for gated breadcrumb links)

Reuses existing tokens for accent (`--accent`), warning border (`--warn-fg` from v1.0.8), and theme-aware status colors (`--success-fg`, `--error-fg`).

### Responsive

`@media (max-width: 800px)` collapses Step 1's two-column grid to single column. Collapse order: Source & paths card on top, Build options card below. Defense in depth — WebView2 window default is 1100×750 per `MainWindow.xaml`; users normally won't hit this.

## Testing

## Implementation phasing

Branch: `feat/v1.0.9-ui-redesign-steps-1-3` off `main`. All commits land sequentially on this branch (single-threaded — same `ui/app.js` + `ui/style.css` files; parallel agents would clobber per `feedback_parallel_agent_conflicts.md`).

### Phase 0 — prep commits (no visible change)

- **A.** `feat(launcher): inject autoScratchPath` — `AutoScratchPath.cs` helper + xUnit tests + `MainWindow.xaml.cs` injection. JS not yet consuming. xUnit 121 → 124.
- **B.** `refactor(ui): factor canMoveForward + add outputPathIsAuto state` — pure logic refactor in `ui/app.js`. Extract gating predicate; add `state.outputPathIsAuto` default; replace filename-string-matching in `syncOutputFilenameToMode`/`prefillOutputIfEmpty` with the boolean check.

### Phase 2 — Step 1 redesign

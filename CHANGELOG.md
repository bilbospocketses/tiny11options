### Fixed
- WPF SDK's auto-generated `Main` collided with `Program.Main` (CS0017). Resolved with `<StartupObject>Tiny11Options.Launcher.Program</StartupObject>` in csproj.
- `Program.Main` now calls `app.InitializeComponent()` before `app.Run()`. Without it, `App.xaml`'s `StartupUri` was never set and the process sat invisible with no window.
- `tiny11maker-from-config.ps1` was committed with the implementation plan's sketch function names (`Import-Tiny11Catalog`, `Resolve-Tiny11Selections -Catalog/-Selected`, `Get-Tiny11VolumeForImage -ImagePath`, `Test-Tiny11SourceIsConsumer -ImageInfo/-ImageIndex`, `Resolve-Tiny11ImageIndex -ImageInfo/-EditionName`, `Invoke-Tiny11Build`). None of these names exist in the actual modules. Rewrote against real exports: `Get-Tiny11Catalog`, `New-Tiny11Selections` + `Resolve-Tiny11Selections -Selections [hashtable]`, `Mount-Tiny11Source` / `Get-Tiny11Editions` / `Dismount-Tiny11Source`, `Test-Tiny11SourceIsConsumer -Editions`, `Resolve-Tiny11ImageIndex -Editions/-Edition`, `Invoke-Tiny11BuildPipeline`. Scaffold-correct only — runtime-validated end-to-end in Phase 7 manual smoke against a real ISO.
- Namespace collisions in handlers + UpdateNotifier (some types fully qualified `global::Tiny11Options.Launcher.Gui.Catalog.Catalog`, others used short names) consolidated.
- Bridge payload-shape mismatch in `ui/app.js`. `BridgeMessage` serializes as `{"type": "...", "payload": {...}}` (per `Bridge.cs` + `BridgeMessage.cs` `[JsonPropertyName]` decorations), but the existing JS handlers for `iso-validated` / `iso-error` / `browse-result` / `profile-loaded` / `build-progress` / `build-complete` / `build-error` / `profile-saved` / `handler-error` were reading their fields directly off `msg` (e.g. `msg.editions`, `msg.message`). Every read was returning `undefined`. Fix: each `onPs` callback now destructures `const p = msg.payload || {}` and reads from `p.<field>`. `state.progress` and `state.completed` now stash `msg.payload` (not the wrapper) so the existing `renderProgress` / `renderComplete` field accesses keep working unchanged. Bug never surfaced in v0.1.0 / v0.2.0 because Path C's bridge buttons hadn't been smoke-tested end-to-end yet.
- Phase 3 bridge contract drift — JS→C# direction (commit `83de5af`, WIP snapshot). Phase 3 was built by 4 parallel agents from plan slices in isolation; agents wrote handlers against the abstract spec, not against the JS-side contract. ~9 message types diverged: `browse-output` / `browse-iso` / `browse-scratch` (renamed to canonical `browse-file` / `browse-folder` / `browse-save-file`), `cancel` (renamed `cancel-build`), `save-profile-request` / `load-profile-request` (renamed `save-profile` / `load-profile` and converted to a 2-step flow with a `pendingSaveProfileSelections` stash), `validate-iso` flat-payload (now nested `payload:{path}`), `apply-update` flat (now nested `payload:{}`), `open-folder` / `close` (payload-wrapped). `IsoHandlers.cs` payload reader switched from `payload?["isoPath"]` (plan-document name) to `payload?["path"]` (JS-side actual). `UpdateHandlers.cs` `HandledTypes` extended with `request-update-check` for the JS-initiated handshake (see "Update-available race fixed" below).
- WebView2 receive: `MainWindow.xaml.cs` `WebMessageReceived` now uses `e.WebMessageAsJson` instead of `e.TryGetWebMessageAsString()`. JS calls `window.chrome.webview.postMessage(<object>)` (idiomatic) rather than pre-stringifying; `TryGetWebMessageAsString` would throw `ArgumentException: Value does not fall within the expected range` on every inbound message.
- Velopack `GithubSource` URL crash on debug launch. `GithubRepoForUpdates` was set to the slug `"bilbospocketses/tiny11options"`. Velopack's `GithubSource` ctor calls `new Uri(repoUrl)` and enforces `Host == "github.com"`; the slug threw `UriFormatException` and silently broke launch-time update checks. Changed to full URL `"https://github.com/bilbospocketses/tiny11options"`.
- Update-available race fixed via JS-initiated handshake (commit `81f8880`). Removed the post-Navigation `_updateNotifier.CheckAsync()` `Task.Run` from `MainWindow.InitializeWebViewAsync`; replaced with a JS-initiated `request-update-check` send from a `DOMContentLoaded` handler in `ui/app.js`. The JS-initiated trigger guarantees the `chrome.webview.addEventListener('message', ...)` listener exists before C# posts the response, eliminating the WebView2 timing race that hid the badge in the prior smoke session.
- `UpdateNotifier.CheckAsync` refactored from push-based to pull-based. Was: `async Task` invoked via `Task.Run`, calling `_bridge.SendToJs(...)` from a thread-pool thread → `MessageToJs` event → `Dispatcher.Invoke` → `PostWebMessageAsString`. Now: `Task<Bridge.BridgeMessage?>` returning the message synchronously to `UpdateHandlers`, which routes through `DispatchJsonAsync`'s return-value path → `SendJsonToJs`, the same delivery path `validate-iso` uses successfully. Avoids the silent-drop failure mode of the prior async-push path (root cause never fully isolated; pull-based design sidesteps it).
- `ui/app.js` `browse-result` handler: now reads `p.context` (was `p.field` — `BrowseHandlers.cs:62` returns `context`); embedded `validate-iso` re-call now payload-wrapped; null-guard early-return on dialog cancel; new branches for `context === 'profile-save'` / `'profile-load'` complete the 2-step flow.
- `IsoHandlers.cs` error-message handling: now parses the script's stdout JSON contract (`{"ok": false, "message": "..."}`) regardless of exit code. Was reading `result.Stderr.Trim()` which is empty when the script exits via its catch block (PS errors written via `[Console]::WriteLine` go to stdout). Falls back to stderr / stdout-as-text / exit-code only when stdout isn't parseable JSON.
- `tiny11-iso-validate.ps1` `Join-Path` 3-arg → 2-arg. PS 5.1's `Join-Path` doesn't support `-AdditionalChildPath` (PS 7+ feature); `powershell.exe` (per dependency-policy waiver) errored with "A positional parameter cannot be found that accepts argument 'Tiny11.Iso.psm1'" and the script never reached its `try` block.
- `tiny11-iso-validate.ps1` `$_.Architecture` property access removed. Same bug + fix as v0.1.0 polish-bundle (commit `cf80091`); Path C scaffold reintroduced it. `Set-StrictMode -Version Latest` trips on missing properties; `$_.Architecture` is unreliable across Windows ISOs. `$_.ImageSize` also dropped to match legacy emit-only-`{index, name}` exactly.
- `launcher/app.manifest`: added `<requestedExecutionLevel level="requireAdministrator" />`. `Mount-DiskImage` + DISM mount + hive-load all require admin; default `asInvoker` failed when launched outside an admin shell. Legacy v0.1.0 inherited admin from already-admin pwsh; the .exe must self-elevate to honor the same dependency.

#### Fixed
- **B1 catalog injection (smoke-blocker, round 2 `93c8a40`)** — see Added entry above.
- **B2 `close` + B3 `open-folder` handlers (round 2 `93c8a40`)** — see Added entry above. JS's "Close" and "Open output folder" buttons now actually do their jobs.
- **B4 catalog path bug (round 3 `b569668`)** — `tiny11maker-from-config.ps1` was passing the catalog DIRECTORY to `Get-Tiny11Catalog -Path`; the function expects the JSON FILE path. `Get-Content -Raw <directory>` throws "Access denied" on Windows, which the launcher's stdout forwarder didn't surface to JS (a separate bug, see thread-affinity below). Now matches `tiny11maker.ps1:126`.
- **B8 SaveFileDialog InitialDirectory (round 4 `0f29e9d`)** — legacy used `<repo>/config/examples` which doesn't extract in Path C builds. Replaced with persisted `UserSettings.LastProfilePath` → MyDocuments fallback. Strictly an upgrade in UX over the legacy hard-coded path.
- **B9 profile catalog validation (round 4 `0f29e9d`)** — see `tiny11-profile-validate.ps1` Added entry. Spawned via `PwshRunner` from `ProfileHandlers.LoadAsync`; no validation logic duplicated in C#.
- **`MainWindow.WebMessageReceived` defensive try/catch (round 4 `0f29e9d`)** — outer handler was bare. `Bridge.DispatchJsonAsync` catches handler-level exceptions, but anything thrown OUTSIDE that path (Serialize failures, MessageToJs subscriber throws, WebView2 disposal mid-dispatch) would land as unobserved event-handler exceptions. Now wrapped: any top-level throw is serialized to `handler-error` and routed back to JS.
- **Resource-cache invalidation via manifest hash (round 4 `0f29e9d`)** — `ExtractUiResourcesIfNeeded` / `ExtractRuntimeResourcesIfNeeded` previously used a content-less `.extracted` marker; adding a new embedded resource (e.g. `tiny11-profile-validate.ps1`) wouldn't trigger re-extraction so the launcher would call PwshRunner against a non-existent script. Caches now compute SHA256 of the sorted resource-name list and write it to `.extracted`; mismatch triggers a wipe + re-extract. New embedded resources auto-invalidate during dev. Caveat: content changes to existing resources without name changes still need a version bump to invalidate (acceptable trade — per-build content hashing would slow every launch; release builds bump version naturally).
- **`SendJsonToJs` WPF thread-affinity violation (`b64c8e1`, root cause behind every silent-async-push failure this session)** — `WebView` is a generated XAML name pointing at a WPF `DependencyObject`. Reading `.CoreWebView2` checks thread affinity and throws `InvalidOperationException: The calling thread cannot access this object because a different thread owns it` from any non-UI thread. Pre-fix `SendJsonToJs` did `WebView?.CoreWebView2 is null` OUTSIDE the `Dispatcher.Invoke` lambda — UI-thread callers (validate-iso, profile-saved) worked fine; build forwarder's `Task.Run` thread blew up before Dispatcher.Invoke ever marshaled, throw landed inside the unobserved Task and was silently swallowed. Same async-push failure that hid the update badge in `81f8880` (worked around there by pull-based refactor; here we keep push and fix the actual root cause). One-line fix: move the entire property+method-call chain INSIDE Dispatcher.Invoke.
- **`BuildHandlers.ForwardJsonLine` defensive DeepClone (round 3 `b569668`)** — `node["payload"].AsObject()` returns a JsonObject still parented under the parsed root. System.Text.Json may throw on re-serialize when the node is referenced from a sibling tree. DeepClone before assigning to a new BridgeMessage; try/catch + smoke-log around `SendToJs` so any future failure logs instead of disappearing.

#### Investigation methodology lessons (logged to memory)
- Round-1 audit's "two-axis enumeration" claim was overclaim — I'd only walked half the legacy entrypoint script. Round 3 walked `tiny11maker.ps1:1-303` end-to-end and the wrapper line-by-line against the cited audit-reference range.
- "Empirically verified during the v0.1.0 investigation" in the v0.2.0 commit message was the lazy framing of "we hit an error during one VL session and stopped hitting it after changing ISOs," which is correlation under multi-bug confusion (the pwsh-from-pwsh bug was active during the same period). ### Diagnostic-stub revert + UpdateNotifier test refactor (2026-05-09)

After smoke check 6 passed end-to-end against a real retail multi-edition Win11 25H2 ISO (full builds completed; cancellations behaved cleanly), the four smoke-only diagnostics introduced during the bridge-debugging session are now reverted to production shape.

#### Removed
- `launcher/App.xaml.cs` `OnStartup` crash logger — the `DispatcherUnhandledException` / `AppDomain.UnhandledException` / `TaskScheduler.UnobservedTaskException` handlers that funneled to `%LOCALAPPDATA%\tiny11options\smoke-crash.log` plus the `MessageBox` UI.
